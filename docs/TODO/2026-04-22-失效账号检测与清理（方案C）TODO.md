## 可解决性评估（本会话补充）

1. **总体判断**：可解决性高（中低复杂度，偏整合型改造）；
2. **关键前提**：
   - 失效判定只针对 `invalid_grant / AADSTS70000` 主口径；
   - 默认治理动作为“置 inactive”，删除仍走二次确认；
3. **风险点**：误判与误删风险，需要靠判定 helper + 审计 + 前端确认流程兜底；
4. **投入预估**：第一版可在 1~2 个迭代日完成（后端字段扩展 + 前端入口闭环 + 契约测试）。

## 实施路径（本会话补充：你问“你打算怎么解决”）

### Step A：先做后端判定与聚合（最小可用）

1. 在 `refresh.py` 增加失效判定 helper（识别 `invalid_grant / AADSTS70000`）；
2. 扩展刷新完成返回字段（`invalid_token_failed_count/list`）；
3. 保持原字段兼容，旧前端可无感运行。

### Step B：补独立治理接口（闭环核心）

1. 新增 `invalid-token-candidates` 查询接口（最近失败 + 失效判定命中）；
2. 新增 `batch-update-status` 接口，支持批量置 `inactive`；
3. 删除继续复用 `batch-delete`，不改保护逻辑。

### Step C：前端接入治理面板

1. 刷新结束时展示失效摘要与“前往治理”入口；
2. 治理面板默认主动作：批量置 `inactive`；
3. 删除作为次动作 + 二次确认。

### Step D：测试与验收收口

1. helper 判定单测 + 接口契约测试；
2. 前端提示与治理流程契约；
3. 人工验收按导入失效账号场景走全链路。
