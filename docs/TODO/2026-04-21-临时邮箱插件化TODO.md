### T4.5 插件设置入口解耦 + 域名选择泛化（后续优化）

**文件**：`static/js/features/temp_emails.js`、`static/js/features/plugins.js`、`templates/index.html`

**问题现状（修复前）**：
- `plugins.js` 已将第三方插件注入 `#tempEmailProviderSelect`
- 但 `temp_emails.js` 的 `loadTempEmails()` / `onTempEmailProviderChange()` 仍只对 `cloudflare_temp_mail` 启用域名下拉
- 因此第三方插件即使 `get_options()` 返回 `domains`，当前页面也无法手动选域名
- 同时插件的 `config_schema` 仍在插件管理卡片内渲染，混合了“安装 / 卸载 / 应用变更”和“业务配置”两类职责

**建议方向**：
1. 插件管理卡片只保留安装 / 卸载 / 应用变更 / 错误展示
2. 插件运行时配置迁移到对应 Provider 设置区或统一 Provider 设置面板
3. 域名下拉启用逻辑改为基于 `/api/temp-emails/options` 返回的 `domains` / `domain_strategy` 判定，而不是硬编码 provider 名称

**当前状态**：
- [x] 已在本会话完成第一轮前端实现：设置页新增独立插件 Provider 配置面板，插件管理卡片的“配置”入口已改为跳转 / 打开设置，插件配置表单不再内嵌在插件卡片内
- [x] 临时邮箱页域名逻辑已改为按 `/api/temp-emails/options` 返回的 `domains` / `domain_strategy` 判定，并增加按 provider 维度的 options cache 与请求防串号保护
- [ ] 本会话未执行人工验收 / 测试命令，仍待后续回归确认
- [x] 已补独立问题文档：`docs/BUG/2026-04-21-插件Provider域名选择未泛化与设置入口耦合BUG.md`

## 6. 风险项

| 风险 | 影响 | 缓解 |
|------|------|------|
| `importlib` 加载恶意插件 | 安全 | 安装时安全警告；约束规则白名单 import |
| 热刷新后内存泄漏 | 长时间运行 | 清理 `sys.modules`；定期重启 |
| 插件与项目版本不兼容 | 功能异常 | `min_app_version` 检查；加载失败优雅降级 |
| 内置 provider 注册被误删 | 系统异常 | `_BUILTIN_PROVIDERS` 白名单 + 单测覆盖 |
| Docker volume 权限 | 目录不可写 | 启动时 `mkdir(exist_ok=True)` |
| 测试间 DB 状态污染 | 测试不稳定 | 每个测试前清理 settings 和 temp_emails 表 |
