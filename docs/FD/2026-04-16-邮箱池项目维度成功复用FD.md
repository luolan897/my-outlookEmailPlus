## 1. 功能定义

### 1.1 核心功能

在**长期邮箱**（Outlook / IMAP 等）范围内，为外部邮箱池补齐“项目维度成功复用”能力：

1. 同一 `caller_id + project_key` 下，某邮箱一旦**成功**使用过，后续不再重复分配给该组合
2. 同一邮箱在一个 `caller_id + project_key` 成功后，可被其他 `project_key` **立即**再次领取
3. 不传 `project_key` 时，继续保持旧行为
4. 多 API Key / 多调用方能力继续存在，但不替代 `caller_id`
5. `task_id` 继续作为任务实例标识保留

### 1.2 本期包含

- ✅ 长期邮箱进入项目维度成功复用语义
- ✅ `caller_id + project_key` 联合边界继续保留
- ✅ `success` 后跨 `project_key` 立即复用
- ✅ 同项目只防成功，不防失败
- ✅ 同一时刻只允许一个活跃 claim
- ✅ `project_key` 继续由调用方自行定义并传入
- ✅ 错误信息继续按正常方式返回

### 1.3 本期不包含

- ❌ 一次性临时邮箱接入本语义
- ❌ 平台内建 `project_key` / 项目管理能力
- ❌ 平台内项目使用记录可视化 / 管理 UI
- ❌ 新增“成功但可复用”的新池状态
- ❌ 额外的分配优先级策略（保持现有候选选择思路）

## 2. 行为设计

### 2.1 适用范围

只有同时满足以下条件时，系统才进入本 FD 定义的新语义：

1. 邮箱类型属于第一阶段覆盖范围（长期邮箱）
2. 调用方显式传入 `project_key`
3. 调用方同时显式传入 `caller_id` 与 `task_id`

若不满足上述条件，则继续按旧行为处理。

本文所称第一阶段排除的“临时邮箱”，在技术实现口径上至少包括：

1. `provider='cloudflare_temp_mail'`
2. `account_type='temp_mail'`

### 2.2 成功后的池状态设计

**设计结论**：第一阶段长期邮箱在显式传入 `project_key` 且 `claim-complete(result=success)` 后，账号本地池状态**直接回到 `available`**。

原因：

1. 可以直接满足“其他 `project_key` 立即复用”
2. 不需要引入新的池状态概念
3. 同项目防重改由项目维度成功记录负责，不再依赖全局 `used`
4. 对外返回语义可以与真实落库状态保持一致，避免出现“成功了但返回仍像全局消耗”的认知冲突

### 2.3 项目维度成功记录

系统需要持久化“某邮箱是否已在某个 `caller_id + project_key` 组合下成功使用过”这一事实。

本设计沿用现有项目使用记录承载（如 `account_project_usage` 或等价语义），但其业务含义需要收敛为：

- **成功记录**
- 而不是“只要 claim 过就算已占用”

也就是说：

1. 只有显式 `claim-complete(result=success)` 才写入/确认项目维度成功记录
2. 失败、超时、过期回收、release 均不应形成同项目永久排除
3. 成功记录默认长期有效，不自动过期
4. 即使管理员后续把邮箱手动改回 `available`，该成功记录仍继续生效

### 2.4 同项目与跨项目规则

#### 同项目

- 同一邮箱只要已经在某个 `caller_id + project_key` 组合下形成了可被新语义可靠识别的 success 记录
- 后续该组合再次 `claim-random` 时，应被排除
- 但对“历史 `used` 长期邮箱迁回 `available`”这一迁移场景，第一阶段接受原项目再次拿到一次的例外

#### 跨项目

- 同一邮箱在项目 A 成功后
- 只要后续请求来自**不同的 `project_key`**
- 且当前邮箱没有活跃 claim
- 则允许再次进入候选集合并被领取

#### 同项目失败重试

若某邮箱在某个 `caller_id + project_key` 下只是：

- 领取后 release
- `verification_timeout`
- `lease_expired`
- 其他未形成成功事实的失败链路

则后续同一 `caller_id + project_key` 仍允许再次拿到该邮箱。

### 2.5 并发占用规则

无论 `project_key` 是否不同，同一邮箱在任意时刻只允许存在一个活跃 claim。

这意味着：

1. 项目维度成功复用属于**完成后的生命周期规则**
2. 不改变现有“claim 是排他占用”的基本模型
3. 其他业务方向必须等待当前 claim 完成或释放后，才可能再次领取同一邮箱

## 3. 接口契约

### 3.1 `/api/external/pool/claim-random`

#### 请求要求

- `caller_id`: 继续必填
- `task_id`: 继续必填
- `project_key`: 选填

#### 行为语义

1. 未传 `project_key`
   - 按旧行为处理

2. 传入 `project_key`
   - 对长期邮箱启用项目维度成功复用语义
   - 判断边界继续使用 `caller_id + project_key`

3. 分配顺序
   - 第一阶段不额外引入新优先级策略
   - 不新增“优先新邮箱”或“优先老邮箱”的产品要求

### 3.2 `/api/external/pool/claim-complete`

#### 长期邮箱 + 已传 `project_key` + `result=success`

系统行为应为：

1. 认定该邮箱在当前 `caller_id + project_key` 下已成功使用
2. 写入/确认项目维度成功记录
3. 清理当前 claim 上下文
4. 将邮箱本地池状态恢复为 `available`
5. 允许其他 `project_key` 在后续请求中再次领取

#### 返回值语义

对于“长期邮箱 + 已传 `project_key` + `result=success`”这一覆盖路径：

1. `claim-complete` 的返回 `pool_status` 应直接体现为 `available`
2. 不再对外返回一个与实际状态不一致的全局 `used` 语义
3. 这样接入方可以从单次完成响应中直接理解：该邮箱对当前项目已成功、但对其他项目仍可继续复用

#### 长期邮箱 + 已传 `project_key` + 非 success

系统行为应为：

1. 不写入同项目成功记录
2. 保持失败语义的正常返回
3. 同一 `caller_id + project_key` 后续仍允许再次领取该邮箱

#### 未传 `project_key` 或不在覆盖范围内

继续按旧行为处理，不进入本 FD 定义的新语义。

### 3.3 错误反馈

第一阶段不额外吞掉、隐藏或弱化错误信息。

要求：

1. 请求失败时继续正常返回错误
2. 无法分配时继续正常返回错误
3. 不因为新增项目维度语义而把错误信息静默吞掉
4. 当邮箱是因“同项目成功历史已命中”而不可分配时，第一阶段继续沿用现有通用失败返回，不额外新增专门错误码或专门错误文案

## 4. 数据语义

### 4.1 账号主状态

对于第一阶段覆盖范围内的长期邮箱：

| 场景 | 期望池状态语义 |
|------|----------------|
| claim 成功后 | `claimed` |
| complete(success) 且传了 `project_key` | 回到 `available` |
| complete(success) 但未传 `project_key` | 维持旧行为 |
| release / timeout / expire 等未成功链路 | 允许后续重试 |

### 4.2 项目维度记录

项目维度记录需要满足：

1. 判断键为 `caller_id + project_key + account`
2. 只记录成功事实
3. 默认长期有效
4. 不因邮箱主状态回到 `available` 而消失

### 4.3 输入角色分层

第一阶段继续保留四层输入语义：

1. **API Key**：识别调用方身份
2. **`caller_id`**：业务请求中的显式调用方标识
3. **`project_key`**：业务方向标识
4. **`task_id`**：一次具体任务/执行实例标识

### 4.4 统计语义

对于第一阶段覆盖范围内、且走“长期邮箱 + 已传 `project_key` + success”路径的账号：

1. 因为账号完成后直接回到 `available`，所以不再应被解释为全局 `used`
2. 现有全局池状态统计应与主状态语义保持一致，避免把这类账号继续统计到 `used`
3. 第一阶段先不额外新增“项目成功次数 / 项目成功明细”的专用统计接口或管理面板
4. 项目维度成功事实仍然存在，但其主要职责是参与后续分配判断，而不是在第一阶段扩展新的统计产品面

---

## 5. 兼容与迁移边界

### 5.1 向后兼容

1. 未传 `project_key` 的调用方继续保持旧行为
2. 仍然要求显式传入 `caller_id`
3. 仍然要求显式传入 `task_id`
4. 多 API Key 能力继续存在，但不替代 `caller_id`

### 5.2 第一阶段排除项

第一阶段不纳入：

1. 临时邮箱
2. 平台项目管理
3. 额外项目使用记录 UI
4. 新池状态扩展
5. 额外选择策略优化

### 5.3 历史数据边界

对于第一阶段覆盖范围内的长期邮箱：

1. 历史上已经被打成全局 `used` 的账号，目标上也应纳入新的项目维度复用语义
2. 不应把“只有新产生的 success 才能复用”作为第一阶段产品语义
3. 也就是说，第一阶段默认不希望历史长期邮箱继续永久滞留在旧的全局消耗模型里
4. 第一阶段接受以“释放历史长期邮箱资产复用”为优先，不要求精确保留历史同项目 success 防重
5. 这意味着历史 `used` 长期邮箱迁回 `available` 后，原项目未来再拿到一次也属于第一阶段可接受代价
6. 具体如何迁移、何时迁移、是否一次性迁移，属于后续 TD / 实现设计阶段讨论内容

### 5.4 后台展示边界

第一阶段即使已有后台列表或详情页，也先不要求额外新增“成功历史”展示位。

当前设计接受：

1. 覆盖路径下的长期邮箱在 success 后主状态显示为 `available`
2. 项目维度成功历史第一阶段先主要作为内部分配判断依据
3. 后台不额外为了本需求新增新的可视化入口、独立字段展示或专门运营面板

---

## 6. 验收口径（FD 视角）

1. 长期邮箱在显式传入 `project_key` 且 success 后，能够直接重新回到 `available`
2. 对新语义下已形成可靠 success 记录的邮箱，同一 `caller_id + project_key` 不再重复分配
3. 不同 `project_key` 对已成功邮箱可立即再次领取
4. 失败/超时/release/过期回收后，同一 `caller_id + project_key` 仍可重试
5. 同一时刻仍只允许一个活跃 claim
6. 不传 `project_key` 时，旧行为不被破坏
7. 错误信息继续正常返回
8. 在覆盖路径下，`claim-complete(success)` 的对外 `pool_status` 返回值与真实状态一致，直接体现为 `available`
9. 在覆盖路径下，全局池状态统计不再把该账号解释为 `used`
10. 第一阶段不额外新增项目成功统计接口或管理面板
11. 第一阶段覆盖范围内的历史 `used` 长期邮箱，产品目标上也纳入新语义，而不是永久停留在旧模型
12. 第一阶段即使后台不新增成功历史展示位，主流程语义仍然成立
13. 当不可分配原因来自同项目成功历史命中时，第一阶段继续沿用现有通用失败返回
14. 第一阶段接受历史 `used` 长期邮箱迁回后，原项目可能再次拿到一次该邮箱的迁移代价
15. 第一阶段文档明确区分“稳定态同项目 success 防重”与“历史 `used` 迁移回池时的一次性例外”

---

## 7. 结论

本 FD 将“项目维度成功复用”落成一个**不新增 UI、不新增项目对象、不新增池状态**的第一阶段设计：

- 对长期邮箱生效
- 对显式传入 `project_key` 的链路生效
- `success` 后直接回 `available`
- 同项目防重依赖成功记录
- 跨项目立即复用

这样既能满足新需求，又能最大程度保持现有接口与系统结构的连续性。
