## 1. 总体架构

### 1.1 数据流

```
[插件开发者]
    │  编写 .py 文件（实现 TempMailProviderBase + config_schema）
    │  提交到官方插件源，更新 registry.json
    ▼
[registry.json]
    │  随版本发版，包含插件名、版本、下载地址、sha256
    ▼
[部署用户 - 安装插件]
    │
    ├─ Web UI: 设置页 → 插件管理 → 点击「安装」
    │     │
    │     ▼
    │   POST /api/plugins/install { name, url? }
    │     │  ← 后端：从 registry.json 或自定义 URL 下载 .py 文件
    │     │  ← 保存到 plugins/temp_mail_providers/{name}.py
    │     │  ← 校验 sha256
    │     │
    │     ▼
    │   返回 { success, plugin_name, file_path }
    │
    └─ CLI: python web_outlook_app.py install-provider moemail
          │
          ▼
        同上逻辑

[部署用户 - 配置插件]
    │
    ▼
POST /api/plugins/{name}/config/schema
    │  ← 读取插件声明的 config_schema
    │
    ▼
返回 { fields: [{ key, label, type, required, placeholder, default }] }
    │
    ▼
前端根据 Schema 动态渲染表单
    │
    ▼
POST /api/plugins/{name}/config
    │  ← 保存配置到 settings 表
    │  ← key 格式: plugin.{provider_name}.{field_key}
    │
    ▼
返回 { success }

[部署用户 - 应用变更（热刷新）]
    │
    ▼
POST /api/system/reload-plugins
    │  ← 后端：清空第三方插件注册表

    │
    ▼
GET /api/temp-emails/options
    │  ← 已注册的插件 provider 出现在 options 中
    │  ← 与内置 provider 完全并列
    │
    ▼

    │
    ▼

# 模块级注册表
_REGISTRY: dict[str, type[TempMailProviderBase]] = {}

def register_provider(cls: type[TempMailProviderBase]) -> type[TempMailProviderBase]:
    """类装饰器：将 Provider 注册到全局注册表"""
    name = getattr(cls, "provider_name", "")
    if name and isinstance(name, str) and name.strip():
        _REGISTRY[name.strip()] = cls
    return cls

def get_registry() -> dict[str, type[TempMailProviderBase]]:
    """返回注册表快照"""
    return dict(_REGISTRY)
```

## 3. 后端：工厂改造

### 3.1 插件目录扫描

**文件：** `outlook_web/services/temp_mail_provider_factory.py`

```python
import importlib.util
import logging
from pathlib import Path

logger = logging.getLogger(__name__)

def _load_plugins() -> list[dict]:
    """
    扫描 plugins/temp_mail_providers/ 目录，加载第三方插件。
    返回加载结果列表 [{name, status, error}]。
    """
    from outlook_web.services.temp_mail_provider_base import _REGISTRY

    plugin_dir = Path(__file__).resolve().parents[2] / "plugins" / "temp_mail_providers"
    if not plugin_dir.is_dir():
        return []

    results = []
    for py_file in sorted(plugin_dir.glob("*.py")):
        if py_file.name.startswith("_"):
            continue
        name = py_file.stem
        try:
            spec = importlib.util.spec_from_file_location(f"plugin.{name}", py_file)
            if spec is None or spec.loader is None:
                results.append({"name": name, "status": "failed", "error": "无法加载模块规格"})
                continue
            mod = importlib.util.module_from_spec(spec)
            spec.loader.exec_module(mod)
            results.append({"name": name, "status": "loaded"})
            logger.info("[plugin] 已加载: %s", name)
        except Exception as exc:
            results.append({"name": name, "status": "failed", "error": str(exc)})
            logger.error("[plugin] 加载失败 %s: %s", name, exc)

    return results
```

### 3.2 工厂函数改造

```python
def get_temp_mail_provider(provider_name: str | None = None) -> TempMailProviderBase:
    from outlook_web.services.temp_mail_provider_base import _REGISTRY

    resolved = settings_repo.get_temp_mail_runtime_provider_name(provider_name)
    if not resolved:
        raise TempMailProviderFactoryError(
            "TEMP_MAIL_PROVIDER_NOT_CONFIGURED",
            "未配置临时邮箱 Provider",
        )

    cls = _REGISTRY.get(resolved)
    if cls is not None:
        return cls(provider_name=resolved)

    raise TempMailProviderFactoryError(
        "TEMP_MAIL_PROVIDER_INVALID",
        f"未知的临时邮箱 Provider: {resolved}",
        data={"provider_name": resolved},
    )


def get_available_providers() -> list[dict[str, Any]]:
    """返回所有已注册的 provider 信息（含内置 + 插件）"""
    from outlook_web.services.temp_mail_provider_base import _REGISTRY
    return [
        {
            "name": name,
            "label": getattr(cls, "provider_label", name),
            "version": getattr(cls, "provider_version", "?"),
            "author": getattr(cls, "provider_author", ""),
        }
        for name, cls in sorted(_REGISTRY.items())
    ]
```

## 5. 配置 Schema 规范

### 5.1 Schema 定义

插件通过 `config_schema` 类属性声明配置字段。这是一个 dict，包含 `fields` 数组。

每个字段定义：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `key` | string | 是 | 配置 key，对应 settings 表中的 `plugin.{provider_name}.{key}` |
| `label` | string | 是 | 中文标签，表单渲染用 |
| `type` | string | 是 | 输入类型，取值见下表 |
| `required` | bool | 否 | 是否必填，默认 false |
| `placeholder` | string | 否 | 输入框占位文本 |
| `default` | any | 否 | 默认值 |
| `description` | string | 否 | 字段说明，显示在输入框下方 |

### 5.2 支持的 type 值

| type | 渲染控件 | 说明 |
|------|---------|------|
| `text` | 文本输入框 | 普通文本 |
| `password` | 密码输入框 | 敏感信息，type=password |
| `textarea` | 多行文本 | 多行输入 |
| `number` | 数字输入框 | 整数/浮点数 |
| `select` | 下拉选择 | 需配合 `options` 字段 |
| `url` | URL 输入框 | 带 URL 校验的文本框 |
| `toggle` | 开关 | 布尔值 |

### 5.3 select 类型的扩展字段

当 `type` 为 `select` 时，需额外提供 `options` 字段：

```python
{
    "key": "region",
    "label": "服务区域",
    "type": "select",
    "required": true,
    "options": [
        { "value": "us", "label": "美国" },
        { "value": "eu", "label": "欧洲" },
        { "value": "asia", "label": "亚洲" },
    ]
}
```

## 8. 后端：热刷新实现

### 8.1 热刷新流程

```
POST /api/system/reload-plugins
    │
    ▼
1. 记录当前 _REGISTRY 中非内置的 provider 名称
    │
    ▼
2. 从 _REGISTRY 中移除所有非内置 provider
    │
    ▼
3. 扫描 plugins/temp_mail_providers/ 目录
    │  对每个 .py 文件：
    │    a. importlib 重新加载
    │    b. @register_provider 自动注册
    │
    ▼
4. 返回加载结果 { loaded: [...], failed: [...] }
```

### 8.2 内置 Provider 保护

内置 provider（`cloudflare_temp_mail`、`custom_domain_temp_mail`）在热刷新时**不受影响**：
- 它们通过模块级 import 注册，不会被卸载
- 热刷新只清空和重新加载 `_REGISTRY` 中非内置的条目

保护机制：`_load_plugins()` 中维护一个内置 provider 白名单，跳过这些名称。

### 8.3 卸载时的活跃邮箱检查

卸载插件前，系统需检查该 provider 是否有关联的活跃邮箱：

```python
def check_provider_in_use(provider_name: str) -> dict:
    """
    检查指定 provider 是否正在使用中。
    返回 { in_use: bool, active_count: int, task_count: int }
    """
    db = get_db()
    user_count = db.execute(
        "SELECT COUNT(*) FROM temp_emails WHERE source = ? AND status = 'active'",
        (provider_name,),
    ).fetchone()[0]
    task_count = db.execute(
        "SELECT COUNT(*) FROM temp_emails WHERE source = ? AND status = 'active' AND mailbox_type = 'task'",
        (provider_name,),
    ).fetchone()[0]
    return {"in_use": user_count > 0, "active_count": user_count, "task_count": task_count}
```

**卸载行为：**

| 场景 | 处理方式 |
|------|---------|
| 无活跃邮箱 | 允许直接卸载 |
| 有活跃用户邮箱 | 显示警告：「该插件下有 N 个活跃邮箱，卸载后这些邮箱将无法读取邮件。确认卸载？」用户确认后允许 |
| 有活跃任务邮箱 | 显示错误：「该插件下有 N 个进行中的任务邮箱，请先完成或结束任务后再卸载。」阻止卸载 |

**热刷新时的处理：** 热刷新只影响插件列表的加载/注册，**不**影响已创建的邮箱记录。即使某个 provider 被卸载，temp_emails 表中的记录保留，用户仍可查看历史邮件（只是无法刷新新邮件）。-provider 的选择回退到内置 provider 或上一个可用 provider。

## 9. 设置页共存策略

### 9.1 两套配置区并存

当前已落地的实现是：内置 provider（CF Worker、GPTMail）保留现有的**专用配置区**（设置页中的独立卡片），插件 provider 的 `config_schema` 则渲染在新增的**统一插件 Provider 配置面板**中。

**原因：**

1. 内置 provider 的配置项较多且与业务逻辑深度耦合（如 CF Worker 的域名同步功能），不适合用通用 Schema 表达
2. 现有用户已配置的内置 provider 设置不受影响，无需迁移
3. 插件配置面板与插件管理卡片分离后，生命周期操作与运行时配置的语义边界更清晰

### 9.2 用户视角的配置入口

按本会话继续落地方案 B 后，当前实现已经把“插件生命周期”和“插件运行时设置”拆开：

- **插件管理卡片**：只负责安装 / 卸载 / 刷新 / 错误展示
- **Provider 设置区**：负责该插件自己的 base_url / token / domain / timeout 等运行时配置

因此当前实现与推荐方向已经基本一致：

| Provider 类型 | 当前配置入口 | 推荐配置入口 |
|--------------|-------------|--------------|
| CF Worker | 设置页专用卡片 | 保持不变 |
| GPTMail | 设置页专用卡片 | 保持不变 |
| 插件 Provider | 统一 Provider 设置面板（`#pluginProviderConfigPanel`） | 保持不变 |

Phase 1 设计核对后还进一步确认了一点：插件 Provider 的独立设置区**不需要新增后端配置协议**。当前 `routes/plugins.py` / `controllers/plugins.py` 已经提供 `config/schema`、`config`、`test-connection` 读写链路；设置页重构的重点是**前端承载位置变化**，而不是把插件配置并入 `/api/settings` 的全局保存。

### 9.3 Provider 下拉列表

无论内置还是插件，都统一出现在临时邮箱页面的 Provider 下拉列表中，不做分组。下拉列表数据源从 `get_available_providers()` 统一获取。

但需要明确一个**当前实现限制**：插件 provider 虽然已经能被 `plugins.js` 注入到 `#tempEmailProviderSelect`，`temp_emails.js` 对域名下拉的启用逻辑却仍写死 `cloudflare_temp_mail`。因此：

1. 插件能出现在 Provider 下拉中
2. 插件也能通过 `/api/temp-emails/options?provider_name=...` 返回 `domains`
3. 但临时邮箱页面是否允许手动选域名，当前仍取决于前端硬编码，而不是 `options.domains` / `domain_strategy`

后续应改为：前端根据 provider 的 `get_options()` 返回值动态决定域名选择 UI，而不是按 provider 名称分支。

`renderConfigForm(schema, currentValues)` 根据 `config_schema.fields` 生成 HTML。

### 11.4 Provider 下拉集成

当前工作树的实际实现是由 `plugins.js` 在加载 `/api/plugins` 成功后，调用 `_refreshProviderRadios()` / `_refreshProviderSelect()` 直接把**已安装插件**注入现有 DOM；随后再由 `main.js` / `plugins.js` 协同驱动独立插件配置面板。

具体行为：
1. 设置页 `.provider-radio-group` 在保留 `legacy_bridge` / `cloudflare_temp_mail` 的基础上，追加插件 provider radio。
2. 临时邮箱页 `#tempEmailProviderSelect` 在保留内置 option 的基础上，追加插件 provider option。
3. 当前会话已补齐页面初始加载时的自动注入，避免必须先展开插件管理卡片才能看到插件 provider。
4. 插件 provider 与内置 provider 视觉并列，不额外分组；选中插件 provider 时会显示独立 `#pluginProviderConfigPanel`，而不是继续在插件管理卡片内部展开表单。

### 11.5 本会话已完成的 UX 收口

本会话已完成前端主干改造，当前体验已收敛为：

1. `plugins.js` 负责插件 Provider 注入、独立插件配置面板渲染，以及插件管理卡片的生命周期操作。
2. `main.js` 负责在设置页切换内置 / 插件 Provider 时，显示对应配置面板。
3. `temp_emails.js` 已不再把“可手动选域名”的条件硬编码为 `cloudflare_temp_mail`，而是根据 `/api/temp-emails/options` 返回的 `domains` / `domain_strategy` 决定域名下拉是否可用。
4. 插件管理卡片中的“配置”入口已改为“打开设置”，配置表单不再内嵌在插件卡片里。

因此，当前剩余事项主要是人工验收与回归确认，而不再是前端承载模型本身。
