## 1. 产品背景与目标

### 1.1 背景

当前系统存在两套独立的邮箱管理体系:

1. **邮箱池系统** (`accounts` 表)
   - 支持 Outlook / IMAP 邮箱的生命周期管理
   - 提供 claim / release / complete 机制
   - 对外接口: `/api/external/pool/*`

2. **CF临时邮箱系统** (`temp_emails` 表)
   - 基于 Cloudflare Workers Temp Email 实现
   - 前端UI已完成,后端Provider已对接
   - 数据存储在独立表,**未接入邮箱池**

**核心问题**: 
- CF临时邮箱无法被邮箱池调度使用
- 注册脚本/第三方服务无法通过pool接口动态获取CF临时邮箱
- 两套系统隔离,管理复杂度高

### 1.2 产品目标

**主目标**: 将CF临时邮箱接入邮箱池系统,实现统一调度管理

**具体目标**:
1. 调用方通过 `/api/external/pool/claim-random`（实现路由为 claim-random，语义上称 claim）可动态创建并获取CF临时邮箱
2. CF临时邮箱使用完成后自动删除远程资源(通过CF Worker API)
3. 统一表模型(accounts表),简化架构
4. 保持现有pool接口契约不变,向后兼容

### 1.3 参考案例

**GPTMail API** (https://www.chatgpt.org.uk/2025/11/gptmailapiapi.html)
- 支持动态创建临时邮箱并接入邮箱池
- 使用完成后自动回收
- 接口简洁,按需创建

## 2. 核心需求

### 2.1 邮箱来源策略

**采用方案: 动态创建模式**

| 对比项 | 动态创建 | 预生成池 |
|-------|---------|---------|
| 资源利用率 | ✅ 按需创建,用完即删 | ❌ 需要提前占用CF配额 |
| 管理复杂度 | ✅ 全自动,无需人工干预 | ❌ 需定期检查池容量 |
| 并发能力 | ✅ 理论无上限(受CF限制) | ❌ 受池容量限制 |
| 成本 | ✅ 按实际使用付费 | ❌ 预占资源可能浪费 |

**决策**: 调用方调用 `claim` 时,如果 `provider=cloudflare_temp_mail` 且池中无可用邮箱,则后端自动调用CF Provider创建新临时邮箱。

### 2.2 数据存储策略

**采用方案: 统一表模型 (accounts表)**

**理由**:
1. 邮箱池代码几乎不改
2. 统一的邮箱管理入口
3. 支持混合调度(Outlook + IMAP + CF)
4. 符合现有分层架构设计

**Schema扩展** (v19):
```sql
-- accounts表新增字段
ALTER TABLE accounts ADD COLUMN temp_mail_meta TEXT;
  -- JSON格式,存储CF特有元数据:
  -- {"provider_jwt": "xxx", "provider_mailbox_id": "123", "address_id": "456"}

-- provider字段复用
-- 新增可选值: 'cloudflare_temp_mail'
```

**历史数据处理**:
- `temp_emails` 表保持只读,不删除
- 新创建的CF邮箱写入 `accounts` 表
- 不提供自动迁移脚本(历史数据量小,可手动处理)

### 2.3 生命周期管理

**使用完成后的处理策略**:

| result值 | pool_status | 远程CF邮箱处理 | 说明 |
|----------|-------------|----------------|------|
| `success` | `used` | **自动删除** | 注册成功,释放CF资源 |
| `verification_timeout` | `cooldown` | **保留** | 验证超时,可能需要重试 |
| `provider_blocked` | `frozen` | **保留** | 被封禁,需人工排查 |
| `credential_invalid` | `retired` | **删除** | 凭据失效,无法继续使用 |
| `network_error` | `available` | **保留** | 网络问题,可重新领取 |

**关键规则**:
1. 只有 `result=success` 和 `result=credential_invalid` 时删除远程CF邮箱
2. 删除失败**非阻塞**,记录日志,本地状态仍正常更新
3. 删除操作通过 `temp_mail_provider_cf.delete_mailbox()` 实现

### 2.4 接口改动

**采用方案: 扩展现有pool接口**

#### 2.4.1 `/api/external/pool/claim-random` 扩展（实现路由为 claim-random，语义上称 claim）

**现有接口保持不变,新增可选参数**:

```json
POST /api/external/pool/claim-random
Headers: X-API-Key: your_key
Body: {
  "caller_id": "reg_script_001",
  "task_id": "task_001",
  "provider": "cloudflare_temp_mail",  // ← 新增,可选
  "project_key": "project_A"           // 现有字段
}

Response (成功):
{
  "success": true,
  "data": {
    "account_id": 123,
    "email": "abc123@cf-domain.com",     // ← CF自动生成
    "claim_token": "clm_xxx",
    "provider": "cloudflare_temp_mail",  // ← 标识类型
    "lease_expires_at": "2026-04-09T12:00:00Z"
  }
}
```

**行为**:
1. 如果 `provider` 参数为空或为 `outlook`/`imap`,行为不变(从池中随机领取)
2. 如果 `provider=cloudflare_temp_mail`:
   - 优先从pool中查找 `pool_status=available AND provider=cloudflare_temp_mail` 的记录
   - 如果没有可用,调用 `temp_mail_provider_cf.create_mailbox()` **动态创建**
   - 创建成功后,直接写入 `accounts` 表,状态为 `claimed`
   - 返回邮箱信息给调用方

**向后兼容**: 不传 `provider` 参数时,行为完全不变。

#### 2.4.2 `/api/external/pool/complete` 扩展

**接口签名不变,内部行为扩展**:

```json
POST /api/external/pool/complete
Headers: X-API-Key: your_key
Body: {
  "account_id": 123,
  "claim_token": "clm_xxx",
  "caller_id": "reg_script_001",
  "task_id": "task_001",
  "result": "success",  // ← 根据此值决定是否删除远程CF邮箱
  "detail": "注册成功"
}

Response:
{
  "success": true,
  "data": {
    "new_status": "used"
  }
}
```

**行为扩展**:
1. 更新 `accounts.pool_status` (现有逻辑)
2. **新增**: 如果 `provider=cloudflare_temp_mail` 且 `result` 需要删除远程邮箱,调用CF Provider删除

#### 2.4.3 邮件读取接口

**无需改动,已支持**:

```json
GET /api/external/messages/latest?email=abc@cf-domain.com
GET /api/external/verification-code?email=abc@cf-domain.com
```

**原理**: 
- `mailbox_resolver.py` 根据 `accounts.provider` 自动识别类型
- `provider=cloudflare_temp_mail` 时,自动调用 `temp_mail_provider_cf.list_messages()`
- 验证码提取逻辑统一,与邮箱类型无关

### 2.5 前端UI需求

**原则: 最小化UI改动,CF邮箱自动化管理**

#### 2.5.1 账号管理页面

**改动**:
- CF邮箱**自动显示**在账号列表
- `provider` 列显示 "Cloudflare 临时邮箱"
- **只读显示**,不支持编辑
- 删除按钮**禁用**(生命周期自动管理)
- 可查看 `pool_status` 和 claim历史

**不改动**:
- 不新增"手动创建CF邮箱"功能(通过pool接口自动创建)

#### 2.5.2 设置页面

**改动**:
- **无需新增**配置项
- 复用现有"CF Worker 配置"区域 (base_url, admin_key, domains)
- 添加说明文字: "配置后,邮箱池可自动创建CF临时邮箱"

## 3. 角色与用例

### 3.1 角色定义

| 角色 | 说明 |
|------|------|
| **注册脚本** | 通过 pool 接口动态获取CF临时邮箱,完成注册任务 |
| **第三方服务** | 集成本系统pool接口,按需获取一次性临时邮箱 |
| **管理员** | 在设置页配置CF Worker,查看账号列表 |

### 3.2 核心用例

#### UC-01: 注册脚本动态获取CF临时邮箱

**优先级**: P0  
**角色**: 注册脚本  
**前置条件**: 管理员已配置CF Worker (base_url + admin_key + domains)

**主流程**:
1. 注册脚本调用 `POST /api/external/pool/claim-random` (provider=cloudflare_temp_mail)
2. 系统检查pool中是否有可用CF邮箱
3. 无可用 → 系统调用CF Provider创建新邮箱(随机前缀 + 默认域名)
4. 系统将邮箱写入 `accounts` 表 (provider=cloudflare_temp_mail, pool_status=claimed)
5. 系统返回 `{email, claim_token}` 给注册脚本
6. 注册脚本使用邮箱完成注册

**结果**: 注册脚本获得一个可立即使用的CF临时邮箱

---

#### UC-02: 获取验证码并完成注册

**优先级**: P0  
**角色**: 注册脚本  
**前置条件**: 已通过UC-01获取CF临时邮箱

**主流程**:
1. 注册脚本在目标平台填写邮箱地址
2. 目标平台发送验证码邮件到CF邮箱
3. 注册脚本调用 `GET /api/external/verification-code?email=abc@cf-domain.com`
4. 系统通过CF Provider读取邮件 (使用meta中的JWT)
5. 系统正则提取验证码 "123456"
6. 系统返回 `{code: "123456"}` 给注册脚本
7. 注册脚本填写验证码,完成注册

**结果**: 注册脚本成功获取验证码

---

#### UC-03: 使用完成并删除远程邮箱

**优先级**: P0  
**角色**: 注册脚本  
**前置条件**: 已通过UC-02完成注册

**主流程**:
1. 注册脚本调用 `POST /api/external/pool/complete` (result=success)
2. 系统更新 `accounts.pool_status = used`
3. 系统识别 `provider=cloudflare_temp_mail` 且 `result=success`
4. 系统调用 `temp_mail_provider_cf.delete_mailbox()` 删除远程CF邮箱
5. CF Provider调用 `DELETE /admin/delete_address/{id}` (使用meta中的address_id)
6. 系统返回 `{success: true, new_status: "used"}`

**结果**: 
- 本地账号状态更新为 `used`
- 远程CF邮箱已删除,释放资源

## 5. 非功能性需求

### 5.1 性能

| 指标 | 目标 |
|------|------|
| claim接口响应时间 | < 3s (含CF Worker创建邮箱) |
| complete接口响应时间 | < 2s (含CF Worker删除邮箱) |
| 并发创建CF邮箱 | 支持10+ QPS |

### 5.2 可靠性

| 场景 | 处理策略 |
|------|----------|
| CF Worker创建邮箱失败 | 返回明确错误码,不写入accounts表 |
| CF Worker删除邮箱失败 | **非阻塞**,记录日志,本地状态仍更新 |
| CF Worker配额耗尽 | create_mailbox返回错误,提示管理员 |

### 5.3 安全性

| 要求 | 实现 |
|------|------|
| CF JWT安全存储 | 明文 JSON 存储在 `temp_mail_meta` 字段（JWT 已签名，本期不额外加密） |
| API Key鉴权 | 复用现有 `/api/external/*` 鉴权机制 |
| pool_access权限 | 复用现有 `external_api_keys.pool_access` 控制 |

## 6. 技术约束

### 6.1 已有约束

1. **SQLite单连接**: 使用 `create_sqlite_connection()` 独立连接,避免跨请求污染
2. **分层架构**: Route → Controller → Service → Repository,禁止跨层
3. **存储策略**: `temp_mail_meta` 以明文 JSON 存储（JWT 已签名，本期不引入额外加密复杂度；后续可增强对 provider_jwt 单独加密）

### 6.2 新增约束

1. **CF Provider调用超时**: 30秒 (已在 `temp_mail_provider_cf.py` 定义)
2. **删除失败重试**: 不重试,仅记录日志(避免阻塞complete接口)
3. **邮箱前缀生成**: 随机8字符小写字母+数字 (复用现有逻辑)

## 7. 风险与应对

| 风险 | 严重度 | 应对策略 |
|------|--------|----------|
| CF Worker配额耗尽 | 中 | create_mailbox失败时返回明确错误码,提示管理员 |
| 删除失败遗留远程邮箱 | 低 | 非阻塞,记录日志,提供手动清理脚本(P2) |
| 并发创建冲突 | 低 | CF Worker内部处理,我们无需关心 |
| temp_emails表历史数据 | 低 | 保持只读,新数据走accounts表,逐步废弃 |
