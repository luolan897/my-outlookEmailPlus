## 1. 文档目标

本 TDD 仅关注本需求的测试设计，不重复业务叙述。

本次测试要证明四件事：

1. 分组策略（范围/正则）可以被正确保存与读取
2. 策略优先级正确：请求参数 > 分组配置 > 默认；组内 regex > length
3. 默认边界正确：仅验证码默认 6 位，链接提取不受影响
4. Web + External 两条链路对同一邮箱/分组策略执行结果一致

---

## 2. 测试分层策略

| 层级 | 目标 | 建议测试文件 | 重点 |
|---|---|---|---|
| A. DB/Repository | groups 规则字段读写正确 | `tests/test_group_verification_policy_repo.py` | length/regex 持久化、默认值 |
| B. Controller(groups) | 分组策略接口契约与校验 | `tests/test_groups_verification_policy_api.py` | 入参校验、错误码、回填 |
| C. 提取链路(Web) | Web 提取策略注入正确 | `tests/test_extract_verification_group_policy.py` | 优先级、默认边界 |
| D. 提取链路(External) | External 同口径行为 | `tests/test_external_verification_group_policy.py` | 与 Web 一致性 |
| E. 前端契约 | 分组弹窗字段与提交体 | `tests/test_group_policy_frontend_contract.py` | modal 字段存在、JS 提交字段 |

执行顺序：A → B → C → D → E。

---

## 3. 测试原则

1. **先契约后逻辑**：先确保分组配置能被正确保存/读取，再验证提取行为。
2. **禁止真实外部模型调用**：AI 相关路径统一 mock。
3. **显式覆盖优先级冲突**：把“regex > length”“请求参数覆盖分组”等写成硬断言。
4. **Web/External 对照测试**：相同输入两条链路结果一致。

---

## 4. 关键测试矩阵

## 4.1 分组策略配置矩阵

| 场景 | 配置 | 预期 |
|---|---|---|
| G-01 | 仅 length=6-6 | 保存成功，读取一致 |
| G-02 | 仅 regex=自定义 | 保存成功，读取一致 |
| G-03 | length + regex 同时配置 | 保存成功，后续提取按 regex 优先 |
| G-04 | 非法 length（如 abc） | 保存失败，明确报错 |
| G-05 | 非法 regex | 保存失败，明确报错 |

## 4.2 提取优先级矩阵

| 场景 | 请求参数 | 分组配置 | 预期 |
|---|---|---|---|
| P-01 | 无 | 无 | 用默认（仅验证码 6 位） |
| P-02 | 无 | length=4-8 | 用分组 length |
| P-03 | 无 | length+regex | 用分组 regex |
| P-04 | code_length=8-8 | 分组 regex | 用请求参数（覆盖分组） |
| P-05 | code_regex=... | 分组 length | 用请求 regex（覆盖分组） |

## 4.3 默认边界矩阵

| 场景 | 输入邮件 | 预期 |
|---|---|---|
| B-01 | 含 6 位码 + 验证链接 | 可提取 6 位码与链接 |
| B-02 | 含 5 位码 + 验证链接 | 默认策略下验证码不命中，但链接可命中 |
| B-03 | 仅验证链接无验证码 | 链接可命中，不受“默认 6 位”影响 |
