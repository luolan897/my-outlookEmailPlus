## 1. 文档目标

本 TDD 仅关注性能优化需求的测试设计，不重复业务叙述。

本次测试要证明六件事：

1. IMAP Token 缓存命中/过期/失效逻辑正确
2. Graph 权限预检能正确解析 scope 并跳过无权限调用
3. IMAP 连接复用组合函数能一次会话完成 fetch + detail
4. 通道缓存写入/读取/过期/失效行为正确
5. IMAP 批量 FETCH 单封/多封解析正确
6. IMAP 并发双服务器竞速逻辑及降级正确

## 2. 测试分层策略

| 层级 | 目标 | 建议测试文件 | 重点 |

| D. 通道缓存 | 写入/读取/TTL/失效 | `tests/test_channel_capability_cache.py` | 缓存行为、过期、手动清除、全量返回 |

执行顺序：A → B → C → D → E → F → G。

## 3. 测试原则

1. **mock 所有外部网络**：MS token endpoint、IMAP 连接、Graph API 全部 mock。
2. **验证调用次数**：连接复用的核心断言是 IMAP 连接只建立 1 次。
3. **时间可控**：token 过期用 `time.monotonic()` mock，不依赖真实等待。
4. **线程安全**：token 缓存和通道缓存需测试并发读写。

## 4. 关键测试矩阵

### 4.1 A 层 — IMAP Token 缓存

| 场景 | 输入 | 预期 |
|---|---|---|
| T-01 首次调用 | 缓存为空 | 请求 MS token endpoint，返回 access_token，写入缓存 |
| T-02 二次调用 | 缓存有效 | 不请求 MS endpoint，直接返回缓存 token |
| T-03 过期后调用 | 缓存存在但 expires_at 已过 | 重新请求 MS endpoint，更新缓存 |
| T-04 不同 refresh_token | 缓存 key 不同 | 各自独立缓存，互不影响 |
| T-05 不同 client_id | 缓存 key 不同 | 各自独立缓存 |
| T-06 手动清除 | 调用 clear_imap_token_cache | 缓存清空，下次调用重新请求 |
| T-07 refresh_token 轮换 | 新 token hash 不同 | 旧缓存不命中，请求新 token |
| T-08 endpoint 返回失败 | 非 200 响应 | 不写入缓存，返回错误 |

### 4.2 B 层 — Graph 权限预检

| 场景 | 输入 | 预期 |
|---|---|---|
| G-01 有 Mail.Read | scope="User.Read Mail.Read profile" | has_mail_read_permission=True，正常调用 Graph |
| G-02 有 Mail.ReadWrite | scope="Mail.ReadWrite User.Read" | has_mail_read_permission=True |
| G-03 无邮件权限 | scope="User.Read profile openid email" | has_mail_read_permission=False，返回 no_mail_permission |
| G-04 scope 为空 | scope="" 或无 scope 字段 | has_mail_read_permission=False |
| G-05 token 刷新失败 | endpoint 返回 error | 正常返回 auth_expired 错误，不检查 scope |
| G-06 scope 大小写 | scope="mail.read" | 不匹配（MS scope 区分大小写） |
| G-07 token 返回含 scope | endpoint 返回 scope 字段 | token_result 包含 scope 字段 |
| G-08 无权限时跳过 Graph API | scope 无 Mail.Read | get_emails_graph 不发起 messages 请求 |
| G-09 有权限时正常调用 | scope 含 Mail.Read | get_emails_graph 正常发起 messages 请求并返回结果 |

### 4.3 C 层 — IMAP 连接复用

| 场景 | 输入 | 预期 |
|---|---|---|
| R-01 有邮件 | 信箱有 3 封邮件 | 返回 emails(top N) + detail(最新 1 封)，IMAP 连接仅 1 次 |
| R-02 空信箱 | 信箱 0 封 | 返回 emails=[], detail=None，IMAP 连接 1 次 |
| R-03 认证失败 | XOAUTH2 authenticate 失败 | 返回 error，无 FETCH 调用 |
| R-04 Token 获取失败 | get_access_token_imap_result 返回失败 | 返回 error，不创建 IMAP 连接 |
| R-05 top=1 优化 | 仅取 1 封 | 摘要和详情来自同一次 FETCH，不重复 |
| R-06 连接异常断开 | 中途 ConnectionResetError | 返回 error，资源正确释放 |

### 4.4 D 层 — 通道缓存

| 场景 | 输入 | 预期 |
|---|---|---|
| C-01 写入可用 | channel=imap_new 成功 | 缓存记录 imap_new 为 available |
| C-02 写入不可用 | channel=graph_inbox 失败 | 缓存记录 graph_inbox 为 unavailable |
| C-03 读取命中 | 已缓存 graph_inbox=unavailable | 返回 unavailable，channel_plan 跳过 |
| C-04 读取未命中 | 新账号无缓存 | 返回 None，channel_plan 正常遍历 |
| C-05 TTL 过期 | 缓存写入 >1h 后 | 返回 None（缓存失效） |
| C-06 手动清除 | token 刷新成功后 | 该账号缓存清空 |
| C-07 不同账号隔离 | 账号 A 缓存不影响账号 B | 各自独立 |
| C-08 无缓存时全量返回 | 全新账号无任何缓存 | filter_channel_plan 返回完整 plan |

### 4.5 E 层 — IMAP 批量 FETCH

| 场景 | 输入 | 预期 |
|---|---|---|
| B-01 单封 | top=1, 1 封邮件 | 正确解析 1 封，等价于逐封 FETCH |
| B-02 多封 | top=5, 5 封邮件 | 一次 FETCH 返回 5 封，全部正确解析 |
| B-03 空结果 | SEARCH 返回 0 个 ID | 不发起 FETCH，返回 emails=[] |
| B-04 部分解析失败 | 5 封中 1 封 RFC822 损坏 | 4 封成功解析，1 封跳过或报错 |
| B-05 大量邮件分页 | top=20 | 一次 FETCH 20 封，不超时 |

### 4.6 F 层 — IMAP 并发双服务器

| 场景 | 输入 | 预期 |
|---|---|---|
| P-01 两台都成功 | server1 2s, server2 3s | 返回 server1 结果（先到先得） |
| P-02 一台成功一台失败 | server1 失败, server2 成功 | 返回 server2 结果 |
| P-03 两台都失败 | 均 auth 失败 | 返回错误 |
| P-04 一台超时一台成功 | server1 timeout, server2 成功 | 返回 server2 结果 |
| P-05 单服务器不并发 | 只传 1 台 server | 直接串行调用，不启线程池 |
| P-06 两台都被调用 | 正常并发场景 | 两台服务器的 fetch 函数都被调用 |

## 15. 真实环境 A/B 性能对照补充（2026-04-12）

### 15.1 目的

在单元/集成测试之外，补充一条可复现实操路径，验证“优化前后”在真实账号上的体感差异。

### 15.2 执行约束

1. 必须使用同一数据库、同一账号、同一接口。
2. 分支切换采用独立后台进程，避免前台阻塞与多实例干扰。
3. 环境固定：
   - `PERF_LOGGING=true`
   - `SCHEDULER_AUTOSTART=false`
4. 每个分支至少 6 次样本。

### 15.3 推荐步骤

1. 启动 `main` 分支实例（后台），采样 6 次。
2. 停止 `main`，启动 `dev-5.3Codex` 实例（后台），采样 6 次。
3. 统计 `avg/min/max`，并对比差值。
4. 同步采集 `[PERF] extract_verification | 总耗时=...` 日志作交叉验证。

### 15.4 本次会话结果（记录）

| 分支 | 样本量 | 平均值 | 最小值 | 最大值 |
|------|--------|--------|--------|--------|
| main | 6 | 6910ms | 5141ms | 9208ms |
| dev-5.3Codex | 6 | 5309ms | 4220ms | 8109ms |

结论：平均耗时下降约 **23.2%**，优化有效；但仍未稳定进入 `≤4s`。

### 15.5 下一步测试补充（Top=3 固定策略）
