# 验证码策略：分组 AI 配置口径错误 + 验证码长度校验易用性问题（BUG）

**创建日期**: 2026-04-10  
**关联模块**: groups 设置弹窗 / groups API / verification 提取链路（Web + External）  
**状态**: 🟢 已修复（2026-04-10）  
**优先级建议**: P1（功能可用性 + 配置闭环）

---

## 1. 问题概述

本次人工反馈确认了两个关联缺陷：

1. **架构口径缺陷（设计/实现偏差）**
   - 当前把 AI 开关与模型 ID 放在“分组配置”中。
   - 但 AI 真正可运行所需的 `Base URL / API Key / Model` 并未在同一配置面板形成闭环。
   - 导致“只看分组配置看似可用，实际并不具备完整可运行配置”。

2. **输入易用性缺陷（可用性 BUG）**
   - 用户配置验证码长度时频繁收到：
     - `GROUP_VERIFICATION_LENGTH_INVALID`
     - `验证码长度范围格式无效`
   - 目前后端仅接受严格 `min-max` 格式（如 `6-6`），对常见输入习惯缺乏容错（如 `6`、`4~8`、`6位`）。

---

## 2. 用户反馈原文关键点（摘要）

1. 分组不应承载 AI 配置。
2. 模型 URL、API Key、模型 ID 属于系统级基础设置，应放设置页。
3. 仅有分组模型 ID 不足以构成可运行 AI 配置闭环。
4. 验证码长度配置“无论怎么设都报错”，实际使用体验不可接受。

---

## 3. 现象与错误信息

### 3.1 已捕获错误

- Message: `验证码长度范围格式无效`
- Code: `GROUP_VERIFICATION_LENGTH_INVALID`
- Status: `400`
- Trace ID: `7fe0755a5e074f919e0da6e26d1c348f`

### 3.2 影响面

1. 分组保存失败，策略无法持久化。
2. 用户对“AI 增强能力可配置且可用”产生误判。
3. 文档与用户真实期望口径偏离，增加沟通和排障成本。

---

## 4. 根因分析（代码事实）

### 4.1 分组 AI 配置口径问题

当前实现中：

- 分组 UI 和 API 包含：
  - `verification_ai_enabled`
  - `verification_ai_model`
- 对应代码位置：
  - `templates/partials/modals.html`
  - `static/js/features/groups.js`
  - `outlook_web/controllers/groups.py`
  - `outlook_web/repositories/groups.py`

但系统级 AI 完整配置（Base URL / API Key / Model）未在同一条“分组策略”路径内闭环，形成能力表达不完整。

### 4.2 长度格式易用性问题

`outlook_web/repositories/groups.py` 中 `_validate_code_length()` 当前仅接受正则：

- `^(\d+)-(\d+)$`

这会拒绝大量用户常见输入：

- `6`
- `4~8`
- `4-8位`
- `6 位`

从而触发 `GROUP_VERIFICATION_LENGTH_INVALID`。

---

## 5. 复现步骤（最小）

### 场景 A：长度格式报错

1. 登录 Web。
2. 打开“添加/编辑分组”。
3. 在“验证码长度范围”输入 `6`（或 `4~8` / `6位`）。
4. 点击保存。
5. 实际返回 400，错误码 `GROUP_VERIFICATION_LENGTH_INVALID`。

### 场景 B：AI 配置闭环缺失

1. 打开分组设置，仅看到 AI 开关 + 模型 ID。
2. 未在同路径看到 AI Base URL / API Key。
3. 用户无法在同一能力配置链路中完成完整 AI 参数配置。

---

## 6. 预期行为

1. **口径预期**：分组只管“验证码规则”（length/regex）。
2. **AI 预期**：AI 参数应在系统设置统一配置（开关、Base URL、API Key、模型 ID）。
3. **长度输入预期**：至少支持常见输入习惯的容错规范化，减少非必要报错。

---

## 7. 修复实施（已完成）

1. groups 去 AI 化（软兼容）
   - 入口：`templates/partials/modals.html`、`static/js/features/groups.js`
   - 后端：`outlook_web/controllers/groups.py`、`outlook_web/repositories/groups.py`
   - 行为：分组仅保留 `verification_code_length` / `verification_code_regex`；历史 `verification_ai_*` payload 接收但忽略，不再参与运行期策略。

2. settings 增加系统级 AI 配置闭环
   - 文件：`outlook_web/controllers/settings.py`、`outlook_web/repositories/settings.py`、`outlook_web/db.py`
   - 新增系统级字段：
     - `verification_ai_enabled`
     - `verification_ai_base_url`
     - `verification_ai_api_key`
     - `verification_ai_model`
   - 行为：API Key 加密存储 + 脱敏回显；AI 开启时若缺 URL/Key/Model，返回 `VERIFICATION_AI_CONFIG_INCOMPLETE`。

3. 提取链路统一改读系统级 AI + 固定 JSON 契约
   - 关键文件：`outlook_web/services/verification_extractor.py`
   - 接入链路：
     - `outlook_web/controllers/emails.py`
     - `outlook_web/services/external_api.py`
     - `outlook_web/services/temp_mail_service.py`
   - 行为：规则优先，低置信度字段再触发 AI fallback；AI 输出 JSON 不合法时快速回退规则结果。

4. 长度输入易用性修复（保持错误码兼容）
   - 文件：`outlook_web/repositories/groups.py`
   - 新增容错规范化：
     - `6` → `6-6`
     - `4~8` / `4～8` → `4-8`
     - `4-8位`、`6 位` → 标准化后校验
   - 非法输入仍返回 `GROUP_VERIFICATION_LENGTH_INVALID`（契约不变）。
