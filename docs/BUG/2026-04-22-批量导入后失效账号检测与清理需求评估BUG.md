# 批量导入后失效账号检测与清理需求评估（BUG/Enhancement）

**创建日期**: 2026-04-22  
**关联 Issue**: https://github.com/ZeroPointSix/outlookEmailPlus/issues/49  
**关联模块**: `outlook_web/controllers/emails.py`、`outlook_web/controllers/accounts.py`、`outlook_web/services/refresh.py`、`static/js/main.js`  
**状态**: 🟡 待实现（会话已确认采用方案 C）  
**优先级建议**: P2（高频运维体验优化）

---

## 1. 问题概述

用户在批量导入 Outlook 账号后，部分账号 refresh token 已失效（典型错误 `invalid_grant / AADSTS70000`）。

当前表现：

1. 在邮件页查看账号时，Graph / IMAP 链路都可能失败；
2. 后端会返回 `EMAIL_FETCH_ALL_METHODS_FAILED`（502）或相关 token 失败错误；
3. 用户缺少“一次性识别失效账号并批量处置”的闭环，清理成本高。

---

## 2. 现状能力（已存在）

系统并非完全无能力，已有以下基础：

1. `POST /api/accounts/refresh-failed`：可重试最近失败账号；
2. `GET /api/accounts/refresh-logs/failed`：可取最近失败列表；
3. `POST /api/accounts/batch-delete`：可批量删除账号；
4. 账号列表已暴露 `last_refresh_status / last_refresh_error` 相关信息。

---

## 3. 缺口定义

当前主要缺口是“能力分散”，缺少针对失效账号治理的统一体验：

1. **检测入口分散**：用户需要手工在多个位置判断哪些账号是失效 token；
2. **处置链路不闭环**：检测后还要手动筛选、再到批量删除；
3. **风险控制不统一**：批量删除属于高风险动作，当前缺少“先标记/再删除”的默认安全路径。

---

## 4. 会话决策（已确认）

本会话已确认采用 **方案 C（混合方案）**：

1. **主入口并入全量检测**：
   - 在用户日常使用的全量刷新/检测链路中直接识别“疑似失效账号”（重点包含 `invalid_grant / AADSTS70000`）。
2. **保留独立清理入口**：
   - 保留可单独操作的“失效账号治理”入口（批量置 inactive / 批量删除），避免强耦合到一次刷新动作里。

---

## 5. 方案 C 的落地原则

1. 默认安全：优先“批量置 inactive”，删除作为二次确认动作；
2. 保持审计：所有批量治理动作都必须记录审计日志；
3. 不破坏现有保护：沿用现有 `cloudflare_temp_mail` 等账号删除保护规则；
4. 错误口径统一：前后端统一识别 `invalid_grant / AADSTS70000`，避免“看起来失败但无法归因”。

## 8. 可解决性深度判断（本会话补充）

### 8.1 结论

**这个 issue 好解决，属于中低复杂度（偏工程整合，不是底层重构）。**

### 8.2 为什么判断“好解决”

已存在的基础能力较完整：

1. 刷新链路本身已经稳定：
   - `stream_refresh_all_accounts()` / `stream_refresh_selected_accounts()` / `refresh_failed_accounts()` 已返回 `failed_list` 与失败信息；
2. 失败数据可直接读取：
   - `GET /api/accounts/refresh-logs/failed` 已有“最近失败账号”数据来源；
3. 处置能力已具备一半：
   - 有批量删除 `POST /api/accounts/batch-delete`；
   - 单账号已有状态更新能力（`active/inactive/disabled`）。

因此本问题核心是“统一判定 + 体验闭环”，而不是从 0 搭新系统。

### 8.3 真实难点（不是没有）

1. 失效判定口径要稳：
   - 不能把所有失败都当作失效；需重点识别 `invalid_grant / AADSTS70000`；
2. 批量处置要安全：
   - 需避免误删，默认推荐“置 inactive”；
3. 前后端字段兼容：
   - 新增统计字段要保持向后兼容，不影响现有刷新 UI。

### 8.4 实施量级预估（文档评估）

1. 后端改动：小到中（新增 helper、扩展返回字段、补一个批量置 inactive 接口）；
2. 前端改动：小到中（失败摘要 + 治理入口 + 二次确认）；
3. 测试改动：中（判定逻辑与契约要补齐）。

**综合评估**：1~2 个迭代日可以落第一版（不含长期观测优化）。

## 9. 拟实施方案（V1 技术草案）

> 本节用于回答“具体怎么改、先改什么、风险怎么控”。

### 9.1 后端方案

1. 在 `outlook_web/services/refresh.py` 增加统一判定 helper（建议命名）：
   - `_classify_refresh_failure(error_message: str) -> dict`
   - 首版聚焦识别：`invalid_grant` / `AADSTS70000`
   - 返回标准字段：
     - `is_invalid_token: bool`
     - `reason_code: str`（如 `INVALID_GRANT_EXPIRED`）
     - `reason_label: str`

2. 扩展刷新结果聚合返回（不破坏旧字段）：
   - 在 `manual_all / manual_selected / retry_failed` 的完成响应中新增可选字段：
     - `invalid_token_failed_count`
     - `invalid_token_failed_list`（受长度上限，避免响应过大）

3. 新增“失效账号候选列表”接口（或增强现有失败日志接口）：
   - 建议：`GET /api/accounts/invalid-token-candidates`
   - 查询口径：账号最近一次 refresh 为 failed，且错误匹配失效判定 helper

4. 新增“批量置 inactive”接口（默认治理动作）：
   - 建议：`POST /api/accounts/batch-update-status`
   - 允许 `active/inactive/disabled`，但治理面板默认仅提交 `inactive`

5. 保持删除接口与保护规则不变：
   - 继续复用 `POST /api/accounts/batch-delete`
   - 保持 `cloudflare_temp_mail` 等保护逻辑与审计记录

### 9.2 前端方案

1. 刷新完成后提示“检测到失效账号 N 个”，提供“前往治理”入口；
2. 新增失效治理面板：
   - 数据源：`invalid-token-candidates`
   - 默认主按钮：`批量置为 inactive`
   - 次按钮：`批量删除`（二次确认输入）
3. 治理后回刷：
   - 刷新失败列表、账号列表与刷新统计，确保界面状态一致。

### 9.3 测试与验收方案

1. 后端单测：
   - 判定 helper 的正负例（包含大小写/多语言/空字符串边界）；
   - 刷新完成响应扩展字段契约测试（旧字段不变）。
2. 接口测试：
   - `invalid-token-candidates` 查询准确性；
   - `batch-update-status` 与 `batch-delete` 在保护规则下的行为。
3. 前端契约：
   - 刷新后提示可见；
   - 治理按钮流程与二次确认流程正确。

### 9.4 风险控制

1. 判定只收敛到 `invalid_grant/AADSTS70000`，避免过度识别；
2. 默认不删除，优先停用（inactive）；
3. 批量删除强制二次确认 + 审计日志。
