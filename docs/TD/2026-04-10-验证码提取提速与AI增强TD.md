## 1. 文档目标

本文档聚焦以下技术实现问题：

1. 分组策略如何在现有分层架构中落地（Route → Controller → Service → Repository）
2. 如何扩展 `groups` 数据模型承载验证码提取策略
3. Web 与 External 两条提取链路如何统一策略解析口径
4. AI 配置如何由系统设置统一承载并在提取链路执行
5. 在不破坏现有接口兼容性的前提下完成演进

---

## 2. 当前技术现状（与本需求相关）

### 2.1 分组能力现状

现有 `groups` 相关链路：

- Route: `outlook_web/routes/groups.py`
- Controller: `outlook_web/controllers/groups.py`
- Repository: `outlook_web/repositories/groups.py`
- DB: `outlook_web/db.py` 中 `groups` 表

当前 `groups` 表字段包含：`name/description/color/proxy_url/is_system/...`，尚未承载验证码提取策略字段。

### 2.2 提取能力现状

主要提取链路：

- Web: `/api/emails/<email>/extract-verification`
- External: `/api/external/verification-code`
- 提取核心：`outlook_web/services/verification_extractor.py`

现有能力支持 `code_regex` / `code_length` 参数化提取，但尚未引入“按分组默认策略”统一注入。

---

## 3. 核心技术决策

### 3.1 决策一：策略存储方式

**选定方案**：`groups` 仅存规则策略（length/regex）；AI 配置放 `settings`（系统级）

**理由**：

1. 分组策略与系统能力职责分离，避免配置语义混乱
2. AI URL/API Key/模型属于系统级基础设置，不适合分组分散维护
3. 保持 group CRUD 简洁，降低误配与排障成本

### 3.2 决策二：分组内规则冲突优先级

**选定方案**：`verification_code_regex` 优先于 `verification_code_length`

**理由**：

1. 正则表达能力更强，覆盖更精准业务场景
2. 与“配置者显式意图优先”的原则一致

### 3.3 决策三：总体策略优先级

**选定方案**：请求参数 > 分组配置 > 系统默认

系统默认限定为：仅验证码默认 6 位，链接提取不受该默认策略影响。

## 4. Schema 迁移设计

## 4.1 目标

在 `groups` 表新增 2 个规则字段承载策略；AI 配置由 `settings` 承载。

## 4.2 新增字段

```sql
ALTER TABLE groups ADD COLUMN verification_code_length TEXT DEFAULT '6-6';
ALTER TABLE groups ADD COLUMN verification_code_regex TEXT DEFAULT '';
```

## 4.3 字段约束语义

1. `verification_code_length`
   - 允许空；非空时形如 `min-max`

2. `verification_code_regex`
   - 允许空；非空时必须可编译

3. 系统级 AI 配置（settings）
   - `verification_ai_enabled`
   - `verification_ai_base_url`
   - `verification_ai_api_key`
   - `verification_ai_model`

## 4.4 迁移幂等要求

1. 迁移前通过 `PRAGMA table_info(groups)` 检查字段存在性
2. 字段已存在则跳过，保证重复执行安全

## 6. Controller 层设计

文件：`outlook_web/controllers/groups.py`

## 6.1 入参扩展

`api_add_group()` / `api_update_group()` 从 request body 解析新增 2 字段（length/regex）。

## 6.2 校验规则

1. `verification_code_length` 非空时校验 `min-max`
2. `verification_code_regex` 非空时校验 regex 可编译
3. 不再承载 AI 字段校验（迁移至 settings 接口）

## 6.3 错误码建议

沿用现有错误响应机制，新增语义化 code（示例）：

- `GROUP_VERIFICATION_LENGTH_INVALID`
- `GROUP_VERIFICATION_REGEX_INVALID`

## 9. 提取链路集成设计

## 9.1 集成原则

不改动现有 API 契约主体，新增“策略解析步骤”注入。

## 9.2 统一策略解析服务（建议）

新增（或抽象）策略解析函数，输入：

- 请求参数（code_regex/code_length）
- 目标邮箱所属 group 配置

输出：

- 最终 `code_regex` / `code_length`
- 系统级 AI 配置（从 settings 读取）

## 9.3 应用位置

1. Web 提取链路（`api_extract_verification`）
2. External verification-code 链路（`api_external_get_verification_code`）

确保两条链路对同一邮箱/分组得到一致策略结果。

## 11.1 兼容性

## 11.2 回滚
