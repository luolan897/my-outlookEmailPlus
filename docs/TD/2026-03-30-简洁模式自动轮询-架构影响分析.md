# 架构影响分析：简洁模式自动轮询

> 分析日期：2026-03-30
> 基于 TD v3 (`docs/TD/2026-03-30-简洁模式自动轮询-TD.md`)
> 分析范围：后端架构、前端架构、数据流、状态管理、事件系统、扩展性

---

## 1. 总体评估

| 维度 | 影响等级 | 说明 |
|------|---------|------|
| 后端架构 | **无影响** | 无新增 endpoint、controller、repository、service，仅扩展 settings 表 3 个 key |
| 数据库 | **无影响** | INSERT OR IGNORE 添加默认值，不需要 schema 版本升级 |
| 前端状态管理 | **中等影响** | 引入双份数据源模式（main.js + mailbox_compact.js），需同步函数 |
| 前端事件系统 | **低影响** | 新增 1 个 CustomEvent，与现有模式一致 |
| 前端文件耦合 | **中等影响** | 跨文件函数依赖 + DOM 结构强耦合 + 渲染函数依赖 |
| 全局命名空间 | **低影响** | 新增 ~15 个函数和 ~5 个变量，与现有模式一致（全局函数） |
| 可扩展性 | **中等影响** | 标准模式轮询无此功能，未来可能需要对齐 |

**结论：本功能不改变现有架构，但引入了 2 个需要注意的架构模式（双份状态源、渲染-轮询耦合）。无需架构层面改动，代码层面做好防御即可。**

## 2. 后端架构分析

### 2.1 无新增组件

本功能**完全复用现有后端架构**：

| 复用组件 | 用途 | 是否修改 |
|---------|------|---------|
| `GET /api/settings` | 加载设置 | 修改：追加 3 个字段到返回值 |
| `PUT /api/settings` | 保存设置 | 修改：追加 3 个字段的校验逻辑 |
| `GET /api/emails/{email}?folder=inbox\|junkemail` | 拉取邮件 | **不修改** |
| `GET /api/emails/{email}/extract-verification` | 提取验证码 | **不修改** |
| `repositories/settings.py` | 设置 CRUD | **不修改** |

### 2.2 不需要新增的组件

- 无新 Controller
- 无新 Repository
- 无新 Service
- 无新 Blueprint
- 无新数据库表
- 无 schema 版本升级（INSERT OR IGNORE 不会覆盖已有值）

### 2.3 对现有后端的影响

**唯一改动点**：`settings.py` 的 GET/PUT 函数各追加 ~15 行代码，遵循已有的 `all_settings.get()` + `queue_setting_update()` 模式，**与现有 50+ 设置字段完全一致**。

**风险评估**：无。改动是纯追加式的，不修改任何现有逻辑。

## 3. 前端状态管理分析

### 3.1 现有状态管理模式

```
main.js 全局变量（30+个）
  ├── accountsCache: {}        ← 账号缓存
  ├── currentGroupId: null      ← 当前分组
  ├── autoPollingEnabled: bool  ← 标准轮询开关
  ├── pollingInterval: number   ← 标准轮询间隔
  └── ... 其他全局变量

标准轮询：所有状态在 main.js 内部闭环
  startPolling() → pollForNewEmails() → stopPolling()
  不跨文件、不跨组件
```

### 3.2 新增状态管理：双份数据源

本功能引入了一个**新的状态管理模式**：

```
main.js                          mailbox_compact.js
┌─────────────────────┐          ┌──────────────────────┐
│ compactPollEnabled  │ ─sync─→ │ compactPollEnabled   │
│ compactPollInterval │ ─sync─→ │ compactPollInterval  │
│ compactPollMaxDuration ─sync─→│ compactPollMaxDuration│
│                     │          │                      │
│ （UI 读写 + 后端持久化）│          │ （轮询引擎运行时读取） │
└─────────────────────┘          └──────────────────────┘
         │                               │
    applyCompactPollSettings() ◄─────────┘
    （main.js 调用，同步到 mailbox_compact.js）
```

**与标准轮询的对比**：

| 维度 | 标准轮询 | 简洁轮询 |
|------|---------|---------|
| 状态位置 | 全部在 main.js | main.js（设置）+ mailbox_compact.js（运行时） |
| 同步机制 | 不需要 | `applyCompactPollSettings()` |
| 竞态风险 | 无 | 理论存在，实际无（单线程 + 同步调用） |
| 运行时状态 | `pollingTimer`, `knownEmailIds` | `compactPollMap`, `compactPollCountdownTimer` |

### 3.3 竞态风险分析

**结论：无竞态风险。**

JavaScript 是单线程的。`applyCompactPollSettings()` 是同步函数调用，在调用返回后所有全局变量已更新。不存在"main.js 更新了一半、mailbox_compact.js 读到不一致状态"的可能。

设置保存流程：
```
用户点击保存 → validateAndSaveSettings()
  → settings.compact_poll_interval = cInterval  // main.js 更新
  → applyCompactPollSettings({ ... })            // 同步到 mailbox_compact.js
  → fetch PUT /api/settings                      // 异步，不阻塞
```

即使 PUT 请求失败，前端状态已经通过 `applyCompactPollSettings` 同步。下次页面加载时会从后端重新读取。

### 3.4 架构建议

当前的双份数据源设计是合理的，因为：
1. `main.js` 需要这些变量来渲染设置面板的初始值
2. `mailbox_compact.js` 需要这些变量来控制轮询行为
3. 跨文件共享全局变量是这个项目的既有模式

**不需要为此引入状态管理库或事件总线**，与项目现有架构保持一致。

## 4. 前端事件系统分析

### 4.1 现有事件使用情况

| 事件类型 | 来源 | 用途 |
|---------|------|------|
| `ui-language-changed` (CustomEvent) | i18n.js | 语言切换后通知所有组件 |
| `click`, `input`, `change` (DOM Event) | HTML inline | 绝大多数交互 |
| `resize` (DOM Event) | main.js | 面板密度调整 |
| `visibilitychange` (DOM Event) | 无 | **当前未使用** |

**关键发现**：整个项目只有 1 个 CustomEvent。本项目的事件系统非常简单，以 inline onclick 为主。

### 4.2 新增事件

| 事件 | 派发者 | 监听者 | 类型 |
|------|-------|-------|------|
| `email-copied` | emails.js | mailbox_compact.js | CustomEvent |
| `visibilitychange` | 浏览器 | mailbox_compact.js | DOM Event |

**`email-copied` 事件**是本功能唯一引入的 CustomEvent。这是继 `ui-language-changed` 之后项目中的第 2 个 CustomEvent。

### 4.3 事件设计评估

**优点**：
- 解耦 emails.js 和 mailbox_compact.js：emails.js 不需要知道轮询引擎的存在
- 遵循观察者模式：未来其他功能也可以监听 `email-copied`（如统计、日志）
- 派发开销极低：不阻塞复制流程

**潜在问题**：
- 如果未来有第 3 个监听者，需要确保监听顺序
- CustomEvent 不支持异步处理，但这里不需要

**结论**：事件设计合理，与项目趋势（从 inline onclick 向 CustomEvent 演进）一致。

## 5. 跨文件依赖分析

### 5.1 依赖关系图

```
emails.js ──── email-copied event ────→ mailbox_compact.js
                                               │
                                         ┌─────┴──────┐
                                         │             │
                                   main.js       groups.js
                                   (showToast)    (copyToClipboard)
                                   (escapeHtml)   (syncAccountSummary
                                   (translateApp   ToAccountCache)
                                    TextLocal)
                                               │
                                         i18n.js
                                   (translateCompactText)
```

### 5.2 依赖调用方式

所有跨文件调用使用 **`typeof` 守卫模式**：

```javascript
if (typeof copyToClipboard === 'function') {
    await copyToClipboard(code);
}

if (typeof syncAccountSummaryToAccountCache === 'function') {
    syncAccountSummaryToAccountCache(email, accountSummary);
}
```

**与现有模式一致**：项目中已有大量 `typeof xxx === 'function'` 检查（如 `groups.js` 调用 `syncAccountSummaryToAccountCache`、`temp_emails.js` 调用各种函数）。

### 5.3 Script 加载顺序要求

```html
<!-- 现有顺序 -->
<script src="/static/js/i18n.js"></script>
<script src="/static/js/main.js"></script>
<script src="/static/js/features/mailbox_compact.js"></script>
<script src="/static/js/features/groups.js"></script>
<script src="/static/js/features/emails.js"></script>
```

**新功能依赖的加载顺序**：
- `mailbox_compact.js` 需要：`i18n.js`（`translateCompactText`）、`main.js`（`showToast` 已在全局）
- `emails.js` 需要：无（CustomEvent 是浏览器原生 API，不需要监听者已加载）
- 运行时需要：`groups.js`（`copyToClipboard`、`syncAccountSummaryToAccountCache`）

**风险评估**：无。CustomEvent 监听者注册在 `mailbox_compact.js` 加载时（脚本解析阶段），而 `email-copied` 事件只在用户交互时派发，远晚于所有脚本加载完成。`copyToClipboard` 等函数通过 `typeof` 守卫保护。

## 6. DOM 耦合分析

### 6.1 轮询引擎依赖的 DOM 结构

轮询引擎通过以下选择器与 DOM 耦合：

```javascript
// 查找账号行
`.pull-button[onclick*="${email}"]` → `.closest('.mail-row')`

// 更新 UI 元素
`.pull-button`           → 按钮文字和样式
`.mail-card-button`      → 脉冲圆点父容器
`.code-button`           → 验证码显示
`.snippet-subject`       → 邮件主题
`.snippet-meta`          → 邮件摘要
```

### 6.2 与 renderCompactAccountList 的耦合

`renderCompactAccountList()`（mailbox_compact.js 第 206 行）全量重建 DOM，会销毁所有轮询指示器。

**解决方案**：在 `renderCompactAccountList()` 末尾调用 `reapplyAllCompactPollUI()` 重新应用轮询状态。

**风险**：
1. 如果有人修改了 `renderCompactAccountList()` 的 DOM 结构（如改了 `.mail-row` 的类名），轮询引擎的选择器会失效
2. 如果有人新增了调用 `renderCompactAccountList()` 的代码路径但忘记调用 `reapplyAllCompactPollUI()`，轮询 UI 会丢失
3. 全量重渲染导致轮询指示器闪烁（先消失再出现）

**缓解措施**：
- `updateSingleRowFromCache()` 做局部 DOM 更新（不触发全量重渲染），减少全量渲染频率
- `uiMissCount` 计数器：连续 5 次 `findCompactAccountRow` 返回 null 时停止轮询，避免无声失败

### 6.3 架构建议

这种 DOM 耦合在当前架构下是不可避免的——项目没有虚拟 DOM 或组件化框架，所有 UI 更新都是直接 DOM 操作。

**建议在代码注释中标注耦合点**：
```javascript
// WARNING: 以下选择器与 renderCompactAccountList() 的 DOM 结构强耦合
// 如果修改 renderCompactAccountList 中的类名，必须同步更新此处
function findCompactAccountRow(email) { ... }
```

## 7. 与现有标准轮询的关系

### 7.1 功能对比

| 维度 | 标准轮询 (main.js) | 简洁轮询 (mailbox_compact.js) |
|------|-------------------|------------------------------|
| 触发方式 | 手动点击"开始轮询" | 自动（复制邮箱后） |
| 适用视图 | 标准模式 | 简洁模式 |
| 轮询单位 | 单个账号（当前选中） | 多个账号（可并行） |
| 停止条件 | 最大次数 / 手动 | 最大时长 / 新邮件 / 手动 |
| 状态存储 | 全局变量 | Map<email, PollState> |
| API | `/api/emails/{email}` | `/api/emails/{email}`（相同） |
| 验证码提取 | 无 | 有（自动） |
| 后台暂停 | 无 | 有（visibilitychange） |

### 7.2 是否需要重构？

**不需要。** 两个轮询系统完全独立：
- 不同的触发方式（手动 vs 自动）
- 不同的视图模式（标准 vs 简洁）
- 不同的状态存储（全局变量 vs Map）
- 不同的停止条件（次数 vs 时长/新邮件）
- 不会同时运行（标准模式和简洁模式是互斥的视图）

**PRD 已明确**（§4.9）："简洁模式轮询与标准模式轮询互不影响，两套轮询逻辑独立运行、独立停止。"

### 7.3 未来的统一考虑

如果将来需要在标准模式也支持"复制后自动轮询"，可以考虑：
1. 将轮询引擎提取为独立模块（`static/js/features/poll_engine.js`）
2. 标准模式和简洁模式共享轮询引擎
3. 通过参数区分行为差异

但这不是当前的需求范围，不建议过度设计。

## 8. 可扩展性分析

### 8.1 未来可能的扩展

| 扩展场景 | 当前设计是否支持 | 需要的改动 |
|---------|----------------|-----------|
| 标准模式也支持自动轮询 | 部分支持 | 需要提取轮询引擎为独立模块 |
| 支持更多轮询触发事件（如右键菜单） | 支持 | 只需在事件监听器中新增入口 |
| 轮询结果推送（WebSocket） | 不支持 | 需要后端新增 WebSocket endpoint |
| 轮询日志/统计 | 不支持 | 需要新增日志收集逻辑 |
| 多浏览器标签页同步 | 不支持 | 需要 BroadcastChannel 或 localStorage event |
| 轮询状态持久化（刷新不丢失） | 不支持 | 需要 sessionStorage/localStorage 持久化 |

### 8.2 扩展性瓶颈

1. **全局函数模式**：轮询引擎作为全局函数定义在 `mailbox_compact.js` 末尾，不利于提取为独立模块。未来如需复用，需要重构加载方式。

2. **DOM 强耦合**：轮询引擎与 `.mail-row` 结构绑定，无法在不同 UI 布局中复用。

3. **设置双份同步**：如果未来设置项继续增长（如轮询策略、通知方式等），双份数据源的模式会变得笨重。

## 9. 性能影响分析

### 9.1 资源消耗

| 资源 | 消耗 | 说明 |
|------|------|------|
| 网络 | 每邮箱每 interval 2 个请求（inbox + junkemail） | 3 个邮箱 × 10s 间隔 = 0.6 req/s |
| CPU | 全局倒计时每秒 1 次 setInterval | 微量，N 个邮箱 × N 次 querySelector |
| 内存 | 每个轮询邮箱 ~200 bytes (PollState) | 可忽略 |
| DOM | 每个轮询邮箱 1 个脉冲圆点 (span) | 微量 |

### 9.2 对现有功能的影响

- **页面加载**：无影响（轮询引擎代码在页面加载后按需运行）
- **设置保存**：无影响（追加逻辑，不改变执行路径）
- **邮件列表渲染**：无影响（标准模式和简洁模式互斥）
- **分组切换**：轮询引擎检查 `isTempEmailGroup` 但不检查分组变化，切换分组后轮询继续运行（合理——用户可能切换回来）

## 10. 安全性分析

### 10.1 无新增攻击面

- **无新 endpoint**：复用现有邮件 API，已有 `@login_required` 保护
- **无新用户输入**：设置面板只有 checkbox 和 number input，后端做范围校验
- **无 XSS 风险**：轮询引擎不使用 innerHTML，所有文本更新通过 `textContent`

### 10.2 注意事项

- `findCompactAccountRow` 中 `email` 直接拼入选择器字符串 `[onclick*="${email}"]`。如果邮箱地址包含 `"` 字符，可能导致选择器注入。**实际风险极低**：Outlook/IMAP 邮箱地址规范不允许出现双引号字符。

## 12. 结论

### 12.1 本功能不需要架构改动

1. **后端**：纯追加式改动，无新组件，无 schema 变更
2. **前端**：遵循现有全局函数 + 全局变量模式，不引入新技术栈
3. **事件系统**：新增 1 个 CustomEvent，与既有 `ui-language-changed` 模式一致
4. **状态管理**：双份数据源有 `applyCompactPollSettings()` 同步，单线程下无竞态

### 12.2 需要关注的架构债务

1. **DOM 强耦合**：轮询引擎与 `.mail-row` DOM 结构绑定，未来 UI 重构可能破坏轮询。建议在代码中标注耦合点。
2. **渲染函数依赖**：`renderCompactAccountList()` 需要在末尾调用 `reapplyAllCompactPollUI()`，这是一个隐式契约，容易被遗漏。
3. **全局命名空间增长**：新增 ~20 个全局标识符（15 函数 + 5 变量），在当前"无模块化"架构下是可接受的，但长期应考虑模块化。

### 12.3 推荐的后续架构优化（非本功能范围）

1. 如果未来标准模式也需要自动轮询，考虑将轮询引擎提取为 `poll_engine.js` 独立文件
2. 如果前端全局变量继续增长（当前 30+），考虑引入简单的状态管理（如 Proxy-based store）
3. 如果 CustomEvent 使用频率增加（当前 2 个），考虑建立统一的事件命名规范（如 `app:email-copied`）
