## 0. 当前状态与范围

当前处于文档设计阶段，本 TD 仅定义第一阶段的技术实现方案，不进入代码实现。

术语约定：

1. **覆盖路径**
   - 指“长期邮箱 + 显式传入 `project_key` + 显式传入 `caller_id` + 显式传入 `task_id`”的处理路径
2. **旧路径**
   - 指“未传 `project_key`”或“不在第一阶段覆盖范围内”的处理路径
3. **第一阶段排除的临时邮箱**
   - 在技术识别上至少包括 `provider='cloudflare_temp_mail'` 或 `account_type='temp_mail'`

本期范围：

1. 长期邮箱在显式传入 `project_key` 时，支持 success 后直接回到 `available`
2. 同项目防重从“claim 痕迹”收敛为“success 记录”
3. 兼容未传 `project_key` 的旧行为
4. 兼容第一阶段“不新增新池状态 / 不新增项目管理 / 不新增项目成功统计面板”的产品边界

本期不处理：

1. 一次性临时邮箱 / `cloudflare_temp_mail` 的项目维度成功复用
2. 新增后台成功历史展示位
3. 新增专门错误码或专门错误文案
4. TDD 落地与代码实现排期

## 1. 技术目标

1. **保持接口兼容**：`claim-random` / `claim-complete` / `claim-release` 入参不新增 `project_key` 以外的新业务字段。
2. **保持状态语义简单**：不新增新池状态，长期邮箱在覆盖路径下 `success` 后直接恢复为 `available`。
3. **收敛项目记录语义**：`account_project_usage` 不再用“是否 claim 过”做永久排除，而是用“是否 success 过”做同项目防重。
4. **兼容历史与新链路并存**：未传 `project_key` 或不在覆盖范围内时，继续按旧状态机工作。
5. **让返回值 / 统计与真实状态一致**：覆盖路径下 `claim-complete(success)` 返回 `available`，全局 `used` 统计不再承载这类账号。

## 2. 当前技术现状

### 2.1 代码锚点

- Repository：`outlook_web/repositories/pool.py`
- Service：`outlook_web/services/pool.py`
- Controller：`outlook_web/controllers/external_pool.py`
- Schema：`outlook_web/db.py`
- 现有测试：`tests/test_pool_flow_suite.py`

### 2.2 当前状态机现状

当前 Repository 层硬编码：

```python
RESULT_TO_POOL_STATUS = {
    "success": "used",
    "verification_timeout": "cooldown",
    "provider_blocked": "frozen",
    "credential_invalid": "retired",
    "network_error": "available",
}
```

也就是说，当前只要 `claim-complete(result=success)`，账号就会被写成全局 `used`。

### 2.3 当前 `project_key` 记录语义

当前 `claim_atomic()` 在 claim 成功时，会直接写入 `account_project_usage`：

- 键：`account_id + consumer_key(caller_id) + project_key`
- 字段：`first_claimed_at / last_claimed_at`

当前问题在于：

1. **claim 时就写入**
   - 这意味着它记录的是“领取痕迹”，不是“成功事实”
2. **release 会删记录**
   - 当前 `release()` 直接按 `account_id + consumer_key` 删除记录，语义仍然是“本次 claim 放弃了，所以删掉 claim 痕迹”
3. **complete 不写 success 记录**
   - `complete()` 只更新 `accounts.pool_status` 与 claim log，不更新 `account_project_usage`

因此，当前 `account_project_usage` 与 PRD/FD 要求的“项目维度 success 记录”并不一致。

### 2.4 当前测试基线暴露的问题

`tests/test_pool_flow_suite.py` 里现有 project 场景测试已经说明真实缺口：

1. `test_claim_with_project_key_prevents_same_project_reuse`
   - success 后需要手工把账号改回 `available`，才能继续验证同项目过滤
2. `test_claim_with_different_project_key_allows_reuse`
   - success 后同样靠手工改回 `available`，才能验证跨项目复用

这说明当前代码并没有原生完成“success 后跨项目继续复用”的生命周期闭环。

### 2.5 当前技术缺口

除了 `success -> used` 之外，还有一个更关键的问题：

**`claim-complete` 当前请求体里没有 `project_key`，而 `accounts` 表里也没有保存“当前 claim 对应的 project_key”。**

这会导致：

1. `claim-random` 时虽然拿到了 `project_key`
2. 但 `claim-complete(success)` 时，系统无法可靠知道“这次成功属于哪个项目”
3. 如果不补 claim 上下文，后续根本无法准确写入“当前项目 success 记录”

因此，本期技术方案不能只改状态流转，还必须补齐**当前 claim 的 `project_key` 上下文存储**。

## 3. 核心技术决策

### 3.1 决策一：覆盖范围判定

**选定方案**：通过账号自身类型判定“是否属于第一阶段长期邮箱覆盖范围”。

建议规则：

1. `provider == 'cloudflare_temp_mail'` → 不进入新语义
2. `account_type == 'temp_mail'` → 不进入新语义
3. 其余 Outlook / IMAP / custom 等长期邮箱 → 可进入新语义
4. 且必须同时满足：
   - 显式传入 `project_key`
   - 显式传入 `caller_id`
   - 显式传入 `task_id`

建议在 Service 层引入统一判断函数，例如：

```python
def _is_project_reuse_eligible_account(row: sqlite3.Row, project_key: Optional[str]) -> bool:
    ...
```

### 3.2 决策二：为当前 claim 增加 `project_key` 上下文

**选定方案**：在 `accounts` 表上新增 `claimed_project_key` 列，保存当前活跃 claim 对应的 `project_key`。

原因：

1. `claim-complete` 当前接口未传 `project_key`
2. 仅靠 `claimed_by=caller_id:task_id` 无法恢复当前项目
3. 把 `project_key` 写进当前 claim 上下文后，`complete/release/expire` 都能在不改接口的前提下拿到正确项目
4. 相比“强行修改 complete API 显式传入 `project_key`”，此方案对接入方破坏更小

行为约束：

1. `claim-random` 成功时：
   - 若本次走项目维度复用语义，则写入 `claimed_project_key`
   - 否则置空 `claimed_project_key`
2. `claim-complete` / `claim-release` / `expire_stale_claims`：
   - 在清理 claim 上下文时同步清空 `claimed_project_key`

### 3.3 决策三：沿用 `account_project_usage`，但把它收敛成“success 记录主载体”

**选定方案**：不新建独立 success 表，继续沿用 `account_project_usage`，但扩展 success 字段，让它同时承载：

1. claim 痕迹（可选保留）
2. success 事实（本期真正参与同项目防重的依据）

建议新增字段：

- `first_success_at TEXT DEFAULT NULL`
- `last_success_at TEXT DEFAULT NULL`
- `success_count INTEGER NOT NULL DEFAULT 0`

收敛后的语义：

1. `first_claimed_at / last_claimed_at`
   - 仅表示历史 claim 痕迹
   - 不能再单独作为“禁止再次领取”的依据
2. `first_success_at / last_success_at / success_count`
   - 才表示真正的同项目 success 事实
   - claim 过滤逻辑只看 success 字段，不看 claim 字段

### 3.4 决策四：`claim-random` 过滤改为只拦“同项目 success 记录”

**选定方案**：保留当前 `NOT EXISTS` 思路，但过滤条件从“存在任意 project usage 记录”改为“存在 success 记录”。

目标查询语义：

```sql
AND NOT EXISTS (
    SELECT 1 FROM account_project_usage apu
    WHERE apu.account_id = a.id
      AND apu.consumer_key = ?
      AND apu.project_key = ?
      AND apu.success_count > 0
)
```

或等价地以 `last_success_at IS NOT NULL` 作为命中条件。

这样可以直接满足：

1. 同项目只有 success 才防重
2. 失败 / release / expire 的 claim 痕迹不会永久挡住后续重试
3. 即使保留历史 claim 行，也不会污染 claim-random 结果

### 3.5 决策五：长期邮箱覆盖路径下，`success` 直接回 `available`

**选定方案**：Repository `complete()` 在覆盖路径下不再套用 `RESULT_TO_POOL_STATUS["success"] = "used"`，而是走专门分支：

1. 写 claim log（`action='complete', result='success'`）
2. 账号主状态更新为：
   - `pool_status='available'`
   - 清空 `claimed_by / claimed_at / lease_expires_at / claim_token / claimed_project_key`
   - `last_result='success'`
   - `success_count = success_count + 1`
3. 同步 upsert `account_project_usage` 的 success 字段

非覆盖路径仍保持旧逻辑：

1. 未传 `project_key`
2. 非长期邮箱
3. 其他未进入项目维度复用语义的链路

这些路径继续使用现有 `RESULT_TO_POOL_STATUS` 映射。

### 3.6 决策六：release / expire / 非 success complete 不再删除项目记录

**选定方案**：`release()` 不再删除 `account_project_usage` 行，只清理 claim 上下文。

原因：

1. claim-random 过滤将改成只看 success 字段
2. 非 success 的行即使保留，也不会阻塞后续同项目重试
3. 删除整行反而会让历史 success / claim 痕迹纠缠，语义更乱

对应行为：

1. `release`
   - 只把账号主状态恢复为 `available`
   - 清空 claim 上下文（包括 `claimed_project_key`）
   - 写 release log
   - 不删 `account_project_usage`
2. `expire_stale_claims`
   - 仍进入 `cooldown`
   - 清空 claim 上下文（包括 `claimed_project_key`）
   - 不写 success 记录
3. `complete(non-success)`
   - 仍按旧结果映射更新主状态
   - 不写 success 字段

### 3.7 决策七：接口返回与统计保持“主状态即真相”

**选定方案**：

1. `claim-complete(success)` 在覆盖路径下返回 `pool_status='available'`
2. `/api/external/pool/stats` 无需新增新接口
3. 由于主状态本身已经回到 `available`，现有 `get_stats()` 的全局 `pool_counts` 会自然体现新语义

这意味着：

1. Controller 层只需继续返回 Service / Repository 计算后的 `new_status`
2. 不需要为第一阶段增加新的统计聚合接口
3. 也不需要在外部 API 里额外暴露“项目成功次数”

## 4. Schema 迁移设计

### 4.1 Schema 版本

**建议升级为 Schema v22。**

原因：

1. 需要给 `accounts` 补充当前 claim 的 `project_key` 上下文
2. 需要给 `account_project_usage` 补充 success 记录字段

### 4.2 `accounts` 表变更

新增字段：

```sql
ALTER TABLE accounts ADD COLUMN claimed_project_key TEXT DEFAULT NULL;
```

字段职责：

- 保存当前活跃 claim 对应的 `project_key`
- 仅在账号 `pool_status='claimed'` 且本次走项目维度复用语义时非空
- 在 release / complete / expire 时清空

### 4.3 `account_project_usage` 表变更

新增字段：

```sql
ALTER TABLE account_project_usage ADD COLUMN first_success_at TEXT DEFAULT NULL;
ALTER TABLE account_project_usage ADD COLUMN last_success_at TEXT DEFAULT NULL;
ALTER TABLE account_project_usage ADD COLUMN success_count INTEGER NOT NULL DEFAULT 0;
```

字段职责：

1. `first_success_at`
   - 第一次在该 `caller_id + project_key` 组合下成功使用该账号的时间
2. `last_success_at`
   - 最近一次成功时间
3. `success_count`
   - 成功次数

兼容策略：

1. 保留现有 `first_claimed_at / last_claimed_at`
2. 不做删列 / 改列名
3. 旧列继续服务于历史痕迹或排查，不再作为 claim 过滤依据

### 4.4 迁移代码约束

`outlook_web/db.py` 中建议遵循现有幂等迁移风格：

1. `PRAGMA table_info(accounts)` 检查 `claimed_project_key`
2. `PRAGMA table_info(account_project_usage)` 检查 3 个 success 字段
3. 已存在则跳过，保证迁移幂等

## 5. Repository 层设计

文件：`outlook_web/repositories/pool.py`

### 5.1 `claim_atomic()` 改造

改造目标：

1. 候选过滤只排除“同项目 success 记录”
2. 领取成功后把当前 `project_key` 写进 `accounts.claimed_project_key`
3. 继续记录 claim log

建议变更：

1. `NOT EXISTS` 子查询增加 success 条件
2. `UPDATE accounts SET ...` 时增加：
   - `claimed_project_key = ?`
3. upsert `account_project_usage` 时：
   - 继续写 `first_claimed_at / last_claimed_at`
   - 不改 success 字段

### 5.2 `complete()` 改造

建议改为：

```python
def complete(
    conn,
    account_id,
    claim_token,
    caller_id,
    task_id,
    result,
    detail,
    *,
    claimed_project_key: Optional[str],
    enable_project_reuse: bool,
) -> str:
    ...
```

覆盖路径下 `result='success'`：

1. `accounts.pool_status = 'available'`
2. 清空当前 claim 上下文（含 `claimed_project_key`）
3. 更新 `success_count / last_result / last_result_detail`
4. upsert `account_project_usage.success_*`
5. 返回 `available`

非覆盖路径：

1. 继续走 `RESULT_TO_POOL_STATUS`
2. 仍然清空 claim 上下文
3. 不写项目 success 字段

### 5.3 `release()` 改造

建议删除当前这段逻辑：

```sql
DELETE FROM account_project_usage
WHERE account_id = ? AND consumer_key = ?
```

改为：

1. 只恢复 `accounts.pool_status='available'`
2. 清空 claim 上下文（含 `claimed_project_key`）
3. 记录 release log
4. 不删除 `account_project_usage`

### 5.4 `expire_stale_claims()` 改造

建议在现有过期流程中补：

1. `claimed_project_key = NULL`
2. 不写 success 字段
3. 不删除项目 usage 行

### 5.5 `get_stats()` 保持不变

由于覆盖路径下账号主状态直接回到 `available`，`get_stats()` 可继续按 `accounts.pool_status` 聚合，不需要专门新增项目成功统计。

## 6. Service / Controller 层设计

### 6.1 Service：`claim_random()`

不改入参，只改内部逻辑：

1. 校验 `caller_id / task_id / project_key`
2. 调 Repository `claim_atomic()`
3. Repository 内部负责记录 `claimed_project_key`

### 6.2 Service：`complete_claim()`

当前 `complete_claim()` 在读取账号时，建议把以下字段一起取出：

- `provider`
- `account_type`
- `claimed_project_key`

然后在 Service 层判断：

1. 本次账号是否属于长期邮箱覆盖范围
2. 当前 claim 是否有 `claimed_project_key`
3. 本次 `result` 是否为 `success`

再把 `enable_project_reuse` 和 `claimed_project_key` 传给 Repository `complete()`。

### 6.3 Controller：`api_external_pool_claim_complete()`

Controller 层无需改接口契约：

1. 请求仍接收当前字段
2. 响应仍返回 `{account_id, pool_status}`
3. 覆盖路径下 `pool_status` 自然变为 `available`

### 6.4 Controller：错误返回

维持现有通用错误返回策略：

1. 同项目 success 历史命中后 claim 失败
2. 继续返回 `NO_AVAILABLE_ACCOUNT`
3. 第一阶段不拆专门错误码

## 7. 历史 `used` 数据迁移策略

### 7.1 当前客观限制

历史数据无法被 100% 精准还原为“哪个 `project_key` 成功过”，原因是：

1. 旧版 `account_claim_logs` 没有记录 `project_key`
2. 旧版 `accounts` 也没有 `claimed_project_key`
3. 旧版 `account_project_usage` 记录的是 claim 痕迹，不是可靠 success 事实

因此，**历史成功事实的 `project_key` 维度无法完全准确回放**。

### 7.2 已确认策略

采用“**主状态先迁移，项目 success 历史不做伪精确回填**”：

1. 仅针对第一阶段覆盖范围内的长期邮箱
2. 将历史 `pool_status='used'` 的账号批量迁回 `available`
3. 不把现有 `account_project_usage` 旧数据直接强行标记成 success
4. 新语义上线后，从新的真实 `claim-complete(success)` 开始累计可靠 success 记录

### 7.3 这样做的收益

1. 能立即兑现“历史长期邮箱不再永久卡死在旧模型里”
2. 不会因为错误回填 success 历史，导致同项目被误伤
3. 实现复杂度可控，且符合第一阶段“先把生命周期跑通”的目标

### 7.4 已接受的代价

1. 历史 `used` 长期邮箱迁回后，原来成功过的项目第一次可能还会再次拿到它
2. 也就是说，**历史数据的“同项目成功防重”会出现一次性弱化**
3. 但从新语义上线后的后续 success 开始，系统会重新建立准确 success 历史

该代价本轮会话已明确接受：

- 对历史数据来说，“以前是不是已经给同项目用过”不再作为第一阶段必须严格保留的迁移目标
- 第一阶段优先级是**释放历史长期邮箱资产，避免继续锁死在旧的全局 `used` 模型里**

### 7.5 备选方案（不推荐）

**方案 B：尝试用旧 `account_project_usage` 直接回填 success**

问题：

1. 它本身是 claim 痕迹，不是 success 事实
2. 会把历史失败 / release / expire 的项目也误记成 success
3. 会造成更严重的同项目误封

因此，不推荐。

## 8. TDD 需要覆盖的测试口径

后续 TDD 至少应覆盖：

1. 长期邮箱 + `project_key` + success → 返回 `available`
2. 同项目 success 后再次 claim → `NO_AVAILABLE_ACCOUNT`
3. 不同项目 success 后再次 claim → 可立即拿到
4. release / verification_timeout / lease_expired 后，同项目仍可重试
5. 未传 `project_key` → 旧行为不变，success 仍进 `used`
6. 临时邮箱 / `cloudflare_temp_mail` → 不进入新语义
7. `claim-complete` 未传 `project_key` 仍能通过 `claimed_project_key` 正确写 success 记录
8. 历史 `used` 长期邮箱迁移后重新进入 `available`

## 9. 风险与注意事项

1. **迁移语义风险**
   - 历史 `used` 账号的项目 success 维度无法被完全准确还原

2. **上下文字段遗漏风险**
   - 若某些路径忘记清理 `claimed_project_key`，会污染后续 complete/release 语义

3. **过滤条件切换风险**
   - `claim-random` 一旦还在看旧 claim 痕迹，就会继续误挡失败重试

4. **特例邮箱混入风险**
   - 若覆盖范围判定不严，临时邮箱可能被错误纳入“success 后回 available”语义

---

## 10. 当前结论

TD 首版已经明确了第一阶段的主技术方向：

1. **主状态层**：长期邮箱 + 显式 `project_key` + success → 直接回 `available`
2. **记录层**：继续沿用 `account_project_usage`，但新增 success 字段，过滤只看 success
3. **claim 上下文层**：必须新增 `accounts.claimed_project_key`，否则 `claim-complete` 无法知道当前成功属于哪个项目
4. **迁移层**：历史 `used` 长期邮箱建议先迁回 `available`，但不做伪精确 success 回填
5. **迁移取舍层**：已明确接受“历史 `used` 长期邮箱迁回后，原成功项目可能再拿到一次”的代价，优先释放历史资产复用

### 11.2 `outlook_web/repositories/pool.py`

   - 非 success

#### E. `expire_stale_claims()`

需要修改：

1. 过期时清理 `claimed_project_key`

#### F. `recover_cooldown()` / `get_stats()`

建议：
