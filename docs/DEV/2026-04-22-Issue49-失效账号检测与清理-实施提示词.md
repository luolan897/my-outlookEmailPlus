## 详细实施步骤（严格顺序执行）

### Phase A：后端判定与结果聚合（最小可用）

1. 在 `refresh.py` 增加 helper（示例命名）：
   - `_classify_refresh_failure(error_message: str) -> dict`
2. 判定规则（V1）：
   - 命中 `invalid_grant` 或 `AADSTS70000` → `is_invalid_token=True`
   - 其他失败不算失效 token
3. 扩展刷新完成响应（可选字段，保持兼容）：
   - `invalid_token_failed_count`
   - `invalid_token_failed_list`（限制长度，如最多 200）

**阶段验收**：
- 旧字段不变，旧前端不报错
- 新字段存在且仅在命中时有值

---

### Phase B：独立治理接口

1. 新增候选列表接口（建议）：
   - `GET /api/accounts/invalid-token-candidates`
   - 返回最近一次 refresh 失败且命中失效判定的账号
2. 新增批量状态接口（建议）：
   - `POST /api/accounts/batch-update-status`
   - 请求：`{ account_ids: number[], status: "inactive" }`
   - 复用单账号状态校验逻辑
3. 删除复用已有：`POST /api/accounts/batch-delete`
   - 不改保护规则

**阶段验收**：
- 候选接口结果准确
- 批量置 inactive 成功并可审计

---

### Phase C：前端闭环

1. 在刷新完成提示里显示：
   - “检测到失效账号 N 个”
2. 提供“前往治理”入口
3. 治理面板动作：
   - 主动作：批量置 inactive
   - 次动作：批量删除（必须二次确认）
4. 治理后自动回刷：
   - 失败列表
   - 账号列表
   - 刷新统计

**阶段验收**：
- 刷新→治理→状态变化可完整走通

---

### Phase D：测试与回归

1. 后端单测：
   - helper 判定（正/负/边界）
   - 刷新扩展字段契约
2. 接口测试：
   - `invalid-token-candidates`
   - `batch-update-status`
   - `batch-delete` 保护行为
3. 前端契约测试：
   - 摘要展示
   - 跳转治理
   - 二次确认
4. 人工验收：
   - 批量导入含失效 token 账号
   - 执行刷新并触发治理闭环
